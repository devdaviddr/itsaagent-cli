# ItsAAgent v0.3.0 — Feature Spec

**Branch:** `docs/v0.3.0-spec`
**Status:** Draft — for review
**Target:** v0.3.0 release (minor — additive, backward-compatible CLI UX)
**Process:** Spec Driven Development — implementation begins only after the spec is approved. Tests are written against this spec.

---

## Vision

v0.2.0 made the agent capable and reliable. v0.3.0 makes it **navigable**: you can see and choose your agent, browse the tools the system actually has, switch agent and model without restarting, and move "back" consistently everywhere. This closes the agent-picker / modal-navigation gap deferred from v0.2.0 — without a TUI rewrite. Everything stays on Clack + the existing Ink run view.

---

## Background

v0.2.0 shipped three built-in agents, user agents, skills, 13 tools, and a home menu — but the menu can't *use* most of it:

1. **Agents can't be chosen interactively.** `--agent` only works at launch; the menu's "Agents" item just prints a list, and Run/Chat from the menu always use `build`.
2. **No mid-session switching.** Once `iaa chat` starts you're stuck on the launch agent. Changing model means quitting and editing config.
3. **Tools are invisible.** There's no way to see what tools exist or what they take, short of reading source.
4. **Navigation is inconsistent.** The menu auto-returns, chat needs `/exit`, sub-prompts use Esc, and a running task only responds to Ctrl+C. There is no uniform "back".

---

## Architecture changes from v0.2.0

| Area | v0.2.0 | v0.3.0 |
|---|---|---|
| Active agent (menu) | Always `build` | Selectable; shown in header; applies to Run/Chat |
| Agent (mid-chat) | Fixed at launch | `/agent <name>` switches it |
| Model (interactive) | Type it in settings | Live picker from `listModels()` |
| Tools | Source only | `iaa tools` command + menu browser |
| Back navigation | Inconsistent | Uniform: Esc = back, "← Back" items, Esc at root = quit |
| `AgentRuntime` agent | `readonly`, set in constructor | `setAgent()` to switch and re-scope tools/prompt |

No new runtime dependencies. No TUI framework change.

## Features

### M-01 · `iaa tools` command and Tools browser

**Motivation:** Users (and the model author) need to see what tools exist and what they accept. Today that requires reading `src/tools/`.

**`iaa tools`** — lists every registered tool: name, one-line description, and required params. Optional `iaa tools <name>` prints full detail (all params with types and descriptions, required vs optional) and which built-in agents permit it.

```
$ iaa tools
Built-in tools (13):
  bash            Execute a bash command on the host system            [command]
  read_file       Read a file (optional start_line/end_line)           [path]
  edit_file       Replace a line range, return a unified diff           [path, start_line, end_line, new_content]
  …
Run `iaa tools <name>` for full parameter detail.

$ iaa tools edit_file
edit_file — Replace lines start_line..end_line (1-indexed, inclusive) …
Parameters:
  path        string   (required)  Absolute or relative path to the file
  start_line  number   (required)  1-indexed first line to replace
  …
Permitted by: build
```

**Menu "Tools" entry** — opens a Clack picker of tool names; selecting one prints its detail, then returns to the Tools list; "← Back" returns to the home menu.

**Acceptance criteria:**
- [x] `iaa tools` lists all registered tools with description and required params
- [x] `iaa tools <name>` prints full parameter detail and which built-in agents permit the tool
- [x] `iaa tools <unknown>` prints a clear "no such tool" message with the list
- [x] Home menu gains a "Tools" entry that browses tools and returns via "← Back"
- [x] Tests: ≥3 — list output contains all tools, detail output for one tool, unknown-tool message

**Files changed:**
- `src/cli/commands/tools.ts` — new `iaa tools [name]` command
- `src/cli/menu.ts` — Tools browser flow
- `src/index.ts` — register `tools` command
- (reuse `getDefaultTools()` and `BUILTIN_AGENTS` for the permitted-by mapping)

### M-02 · Agent selection in the home menu

**Motivation:** The menu should let you pick which agent Run and Chat use, and show the current choice.

**Behaviour:**
- The home menu tracks an **active agent** for the session (default `build`).
- A menu item `Agent: <id> ▸` opens a picker listing all agents (built-in first, then `[custom]`), with description and tool count; the current one is preselected.
- Selecting an agent updates the active agent; **Run a task** and **Chat** then use it.
- The active agent appears in the menu status header (see M-05).
- Selection is session-local (not written to config).

**Acceptance criteria:**
- [ ] Menu shows an `Agent: <id>` item reflecting the active agent
- [ ] Selecting an agent updates it; subsequent Run/Chat use that agent
- [ ] Picker lists built-ins first, then custom agents tagged `[custom]`
- [ ] Esc / "← Back" in the picker leaves the active agent unchanged
- [ ] Tests: ≥2 — picker returns selected id; menu state helper applies it to the run config

**Files changed:**
- `src/cli/menu.ts` — active-agent state, picker, pass agent into `runTaskFlow`/`chatFlow`
- `src/cli/agentSelect.ts` — new: reusable agent-picker helper (also used by M-03 indirectly)

### M-03 · In-chat slash commands (`/agent`, `/agents`, `/model`, `/help`)

**Motivation:** Switching agent or model mid-conversation without quitting is the single biggest workflow win.

**New chat commands** (alongside existing `/exit`, `/clear`):
- `/help` — list available slash commands
- `/agents` — list agents inline
- `/agent <name>` — switch the active agent. Because the system prompt and tool set change, this **re-initialises the session** (context reset) and prints a notice: `Switched to <name> — context cleared.` Unknown name → error, no change.
- `/model <name>` — switch the model for subsequent turns (validates against `listModels()`); persists to config. Unknown model → error, no change.
- The chat prompt shows the active agent, e.g. `build ›` instead of `>`.

**`AgentRuntime.setAgent(def)`** — updates the active agent and re-derives permitted tools; the next `initSession()` rebuilds the system prompt for it.

**Acceptance criteria:**
- [ ] `/help` lists all slash commands
- [ ] `/agent <name>` switches agent, resets context, prints a notice; unknown name errors without changing state
- [ ] `/agents` lists agents inline
- [ ] `/model <name>` switches and persists the model; unknown model errors
- [ ] Chat prompt reflects the active agent id
- [ ] `AgentRuntime.setAgent()` re-scopes permitted tools and the next system prompt
- [ ] Tests: ≥4 — setAgent re-scopes tools, /agent reset behaviour (unit on a parsing helper), unknown agent rejected, slash-command parser maps inputs

**Files changed:**
- `src/agent/AgentRuntime.ts` — `setAgent()`; make the active agent mutable
- `src/cli/commands/chat.ts` and `src/cli/menu.ts` (chatFlow) — slash-command handling
- `src/cli/chatCommands.ts` — new: pure parser/dispatcher for slash commands (testable)

### M-04 · Consistent back navigation

**Motivation:** "Back" should mean the same thing everywhere.

**Rules:**
- Every sub-menu (agent picker, model picker, tools list, settings) includes an explicit **"← Back"** option.
- **Esc / Ctrl+C** in any sub-menu or prompt = go back one level (cancel without changing state).
- **Esc / Ctrl+C at the home menu** = quit (with the existing outro).
- A **running task** still cancels with Ctrl+C; chat still uses `/exit` to return to the menu, and this is shown in the hint line.

**Acceptance criteria:**
- [ ] Each sub-menu has a "← Back" item that returns to its parent without side effects
- [ ] Esc in a sub-menu returns to the parent; Esc at the home menu quits
- [ ] No flow dead-ends or exits the process unexpectedly on cancel
- [ ] Tests: ≥2 — a navigation helper maps cancel → "back" vs "quit" by depth

**Files changed:**
- `src/cli/menu.ts` — back items, cancel handling per level
- `src/cli/menu.ts` helper `handleCancel(depth)` (pure, testable)

### M-05 · Menu status header

**Motivation:** Always show what you're operating with.

**Behaviour:**
- The home menu renders a status line above the choices: `<agent> · <model> · <provider> · <host>`, refreshed after any change (agent/model/settings).
- If Ollama is unreachable when the menu opens, the line shows `⚠ provider unreachable` but the menu still works (existing behaviour, extended with agent/model).

**Acceptance criteria:**
- [ ] Header shows active agent, model, provider, host
- [ ] Header updates after agent/model/settings changes within the session
- [ ] Unreachable provider shows a warning without breaking the menu
- [ ] Tests: ≥1 — header formatting helper renders the expected string

**Files changed:**
- `src/cli/menu.ts` — `statusHeader(state)` helper + re-render after changes

---

### M-06 · Model picker in the menu

**Motivation:** Picking a model from a live list beats typing it into settings.

**Behaviour:**
- A menu item `Model: <name> ▸` opens a picker populated from `OllamaProvider.listModels()`, with a `⚡` marker for models that support native tool use (via `supportsTools()` — best-effort, may be skipped if slow).
- Selecting persists `model` to config and updates the header.
- If the provider is unreachable, the picker shows an error and returns to the menu (no crash).

**Acceptance criteria:**
- [ ] Picker lists live models from the configured provider
- [ ] Selecting a model persists it to `config.json` and updates the header
- [ ] Native-tool-use marker shown where detectable
- [ ] Provider error shows a message and returns to the menu
- [ ] Tests: ≥2 — selection persists via the config helper; unreachable path handled

**Files changed:**
- `src/cli/menu.ts` — model picker flow (reuses `promptForModel` in `src/cli/select.ts` where possible)

## What's deferred (not in v0.3.0)

| Item | Reason |
|---|---|
| Full OpenTUI/SolidJS TUI rewrite | Still chrome over substance; Clack + Ink suffice |
| Tab-to-cycle agents (OpenCode-style) | `/agent` and the picker cover it; a keybind is polish |
| Persisting the menu's active agent to config | Session-local is clearer; revisit if requested |
| Mid-chat context **preservation** across an agent switch | Switching resets context (system prompt changes); preserving it is ambiguous |
| Per-skill selection in the menu | Skills via `--skill` / `/name`; menu selection later |
