# ItsAAgent v0.2.0 — Feature Spec

**Branch:** `feat/v0.2.0-spec`  
**Status:** Draft  
**Target:** v0.2.0 release  
**Process:** Spec Driven Development — implementation begins only after spec is approved. Each feature defines acceptance criteria before a line of code is written. Tests are written against this spec, not against the implementation.

---

## Background

v0.1.0 shipped a working local ReAct agent with 6 tools, Ink TUI, Ollama streaming, and 40 unit tests. The review of v0.1.0 — including an architecture comparison against OpenCode and Crush — identified five categories of work for v0.2.0:

1. **Architecture** — text-based tool parsing is the biggest quality gap vs. native function calling (OpenCode, Crush both use it); Ollama has supported native tool use since late 2024
2. **Agent quality** — parser fragility, missing tool types, context eviction blindness
3. **Developer workflow** — the agent can't do git operations or targeted file edits (only full rewrites)
4. **UX** — non-TTY mode is silent, no history browser, SSH requires env var password
5. **Extensibility** — no external plugin system

### F-01 · `edit_file` tool

**Motivation:** The agent currently rewrites entire files via `write_file`. This destroys git diff readability, is unsafe for large files, and causes conflicts in multi-step edits. A line-range replacement tool lets the agent make targeted changes.

**Behaviour:**
- Takes `path`, `start_line` (1-indexed, inclusive), `end_line` (1-indexed, inclusive), and `new_content`
- Reads the file, replaces lines `start_line..end_line` with `new_content`, writes back
- Returns a unified diff of the change (old vs new) in `data` so the model can verify
- `start_line === end_line` replaces a single line
- `end_line` may equal `start_line - 1` (i.e. end < start) to perform a pure insertion before `start_line`
- Out-of-range lines return `{ success: false, error: "line X out of range (file has N lines)" }`

**Acceptance criteria:**
- [ ] Replaces a range of lines and returns a diff
- [ ] Inserts lines without removing existing content when `end < start`
- [ ] Fails gracefully on missing file
- [ ] Fails gracefully on out-of-range line numbers
- [ ] Does not rewrite the entire file (reads, splices, writes)
- [ ] Tests: ≥5 covering replace, insert, out-of-range, missing file, diff output

**API contract:**
```typescript
definition.name: "edit_file"
parameters.required: ["path", "start_line", "end_line", "new_content"]
// start_line: number — 1-indexed
// end_line: number   — 1-indexed, inclusive
// new_content: string — replacement text (may be empty string to delete lines)
```

### F-02 · `fetch` tool

**Motivation:** The agent cannot access the web. `fetch` unlocks a large class of tasks: reading documentation, downloading files, calling APIs, checking status pages.

**Behaviour:**
- Supports GET and POST
- `url`, `method` (default `GET`), `headers` (optional JSON object), `body` (optional string for POST)
- Follows redirects (max 5)
- Returns: status code, response headers (content-type only), truncated body (max 8 KB)
- For HTML responses, strips tags and returns plain text (via regex strip — no full DOM parser)
- Timeouts after 15 s
- Does not follow `file://` or non-HTTP(S) schemes — returns error

**Acceptance criteria:**
- [ ] GET returns status + truncated body
- [ ] POST sends body with correct Content-Type
- [ ] HTML is stripped to plain text before returning
- [ ] Non-HTTP schemes rejected with clear error
- [ ] Timeout fires at 15 s
- [ ] Tests: ≥4 covering GET, POST, HTML strip, invalid scheme (use `nock` or inline mock)

**API contract:**
```typescript
definition.name: "fetch"
parameters.required: ["url"]
parameters.optional: ["method", "headers", "body"]
// Returns: "HTTP 200\n\n<body up to 8KB>"
```

### F-03 · `git` tool

**Motivation:** Developers use this agent to make code changes. Without git awareness the agent cannot propose commits, check what changed, or work in branches. This is the single highest-leverage tool gap.

**Behaviour:**
- `subcommand`: one of `status`, `diff`, `log`, `add`, `commit`, `branch`, `checkout`
- `args`: string of additional arguments passed to git (e.g. `"--staged"`, `"-m 'fix: typo'"`)
- `cwd`: working directory (defaults to process.cwd())
- Executes via `execFile("git", [subcommand, ...args.split(" ")])` — no shell
- `commit` subcommand requires `args` to contain `-m "..."` — validates message present before running
- Destructive subcommands (`reset`, `push --force`, `clean`) are blocked — return `{ success: false, error: "subcommand not permitted" }`
- Output truncated at 6 KB

**Accepted subcommands:** `status`, `diff`, `log`, `add`, `commit`, `branch`, `checkout`, `show`, `stash`  
**Blocked subcommands:** `push --force`, `reset --hard`, `clean`, `rebase` (for now)

**Acceptance criteria:**
- [ ] `git status` returns stdout
- [ ] `git add <file>` stages a file
- [ ] `git commit -m "msg"` creates a commit
- [ ] Blocked subcommands return error without executing
- [ ] `commit` without `-m` returns error before running git
- [ ] Tests: ≥5 covering status, add, commit, blocked command, missing message

**API contract:**
```typescript
definition.name: "git"
parameters.required: ["subcommand"]
parameters.optional: ["args", "cwd"]
```

### F-04 · Streaming feedback in plain / non-TTY mode

**Motivation:** When output is piped or the terminal isn't a TTY, the agent runs silently for minutes. Users have no feedback and cannot tell if it has hung. `-v` in plain mode should print thoughts and tool calls as they happen, not wait for completion.

**Behaviour:**
- In `renderPlain` (non-TTY path), when `verbose` is true, the `thought` and `tool:call` events already print to stderr
- The gap: `chunk` events are never printed in plain mode, so the model's streaming output is invisible until a `thought` event fires
- Add a `raw` flag to `renderPlain`: if enabled, print `chunk` deltas to stderr with no formatting (for pipe-friendly debugging)
- Add `--raw` CLI flag to `ai run` and `ai chat` — enables plain streaming regardless of TTY state

**Acceptance criteria:**
- [ ] `ai run "task" --raw` prints chunk deltas to stderr as they arrive
- [ ] `--raw` works in both TTY and non-TTY environments
- [ ] `--raw` is documented in `ai run --help`
- [ ] Does not affect the final stdout output (answer still printed to stdout)
- [ ] Tests: unit test that `renderPlain` with `raw: true` emits chunk events to stderr

### F-05 · Context eviction warning

**Motivation:** The agent silently drops messages when the context window fills. If it drops the original task or a key tool result, the next response is incoherent. The user has no way to know this happened.

**Behaviour:**
- `ContextManager` emits a warning string from a new `onEvict?: (count: number) => void` callback registered at construction
- `AgentRuntime` passes a callback that emits a new `context:evict` event: `{ evicted: number; remaining: number; ratio: number }`
- TUI: displays a one-line amber warning in the step list when eviction occurs: `⚠ Context trimmed (N messages evicted, ratio: X%)`
- Plain mode: prints to stderr when verbose

**Acceptance criteria:**
- [ ] `ContextManager` calls `onEvict` with the count of evicted messages
- [ ] `AgentRuntime` emits `context:evict` event
- [ ] TUI renders the amber warning when `context:evict` fires
- [ ] Tests: `ContextManager` calls `onEvict` with correct count after trimming

### F-06 · External tool plugin system

**Motivation:** Users want to add custom tools without modifying the source. A plugin directory lets them drop in a JS file and have the tool available immediately.

**Behaviour:**
- On startup, `getDefaultTools()` scans `~/.config/ai-cli/tools/*.js` (CommonJS or ESM)
- Each file must export a default `Tool` object matching the existing `Tool` interface
- Invalid plugins (missing `definition`, wrong shape) are skipped with a warning to stderr — they never crash the agent
- `ai check` reports loaded plugins: `✓ 2 plugin tools loaded`
- No sandboxing in v0.2.0 — plugins run with full Node.js permissions (document this clearly)

**Acceptance criteria:**
- [ ] A valid `.js` file in `~/.config/ai-cli/tools/` is loaded and usable in a run
- [ ] An invalid plugin (bad export shape) is skipped, not crashed
- [ ] `ai check` counts plugin tools separately from built-in tools
- [ ] Tests: mock the plugin directory, verify valid plugin loads and invalid plugin is skipped

### F-07 · `ai history` command

**Motivation:** Session logs are written to `~/.config/ai-cli/logs/` but there's no way to browse them from the CLI. Users have to `ls` and `cat` manually.

**Behaviour:**
- `ai history` lists the 20 most recent sessions: timestamp, task (first line), step count, model
- `ai history --tail 5` limits to 5
- `ai history <session-id>` prints the full markdown log for that session
- Session ID is the filename stem (`session-2026-06-19_15-08-54`)
- If no logs exist: `No sessions recorded yet. Run with -v or -l to start logging.`

**Acceptance criteria:**
- [ ] Lists sessions sorted newest-first
- [ ] `--tail` limits results
- [ ] Session detail prints raw markdown
- [ ] Empty log dir shows friendly message
- [ ] Tests: mock log directory, verify listing and detail output

### F-08 · Stronger XML parser

**Motivation:** The current parser uses simple regex (`/<thought>([\s\S]*?)<\/thought>/i`). Tool output containing XML-like strings (e.g. HTML in a bash result) can cause false tag matches and mis-parse the response.

**Behaviour:**
- Replace regex-based tag extraction with a minimal linear scanner that:
  1. Finds the first `<thought>` occurrence
  2. Scans forward for the matching `</thought>` (not just the first `</thought>` in the string)
  3. Ensures the extracted content doesn't contain another `<thought>` open tag (if it does, take the innermost)
- Same approach for `<answer>` and `<tool_call>`
- All existing parser tests must continue to pass
- Add new tests: nested tags, tags inside bash output in the raw string, unclosed tags

**Acceptance criteria:**
- [ ] All 10 existing parser tests pass unchanged
- [ ] Tags appearing inside tool output content do not cause mis-parse
- [ ] Unclosed tags fall through to the next fallback (not a crash)
- [ ] Tests: ≥3 new edge-case tests for the above scenarios

### F-09 · Native Ollama tool-use (function calling)

**Motivation:** ItsAAgent's 3-fallback text parser is the single biggest architectural weakness compared to tools like OpenCode and Crush. Both use native function calling via provider APIs — the model emits structured tool invocations, not free-form text containing JSON. Ollama has supported native tool use since late 2024 for compatible models (`qwen2.5`, `mistral`, `llama3.1`, `llama3.2` and others). Adopting it means the API layer enforces structure rather than a fragile parser — closing the majority of the quality gap with cloud-first agents.

**What changes:**
- `OllamaProvider.stream()` sends `tools` in the request body (Ollama's function-calling format, OpenAI-compatible)
- The model responds with `message.tool_calls[]` in the NDJSON stream instead of free-form text
- `AgentRuntime` reads `tool_calls` from the stream response directly — no parsing required
- The 3-fallback `parseResponse()` parser becomes a **fallback for non-tool-use-capable models only** — it fires when the response contains no `tool_calls` field
- Model capability is detected at startup via `OllamaProvider.listModels()` — models with `tools` in capabilities use native calling; others fall back to text parsing
- Tool definitions are translated from the existing `ToolDefinition` schema to Ollama's function schema (they are already OpenAI-compatible)

**Behaviour:**
- On each loop iteration, tools are sent as `tools: [...]` in the Ollama request
- If the response includes `message.tool_calls`, extract `name` and `arguments` directly — no `parseResponse()` call
- If the response has no `tool_calls` (model doesn't support it, or produced a final answer), fall through to the existing `parseResponse()` chain
- `thought` / reasoning content still extracted from `message.content` when present alongside tool calls
- `isExplicitAnswer` is true when response has no `tool_calls` and content matches `<answer>` tag or is plain text with no tool intent

**Acceptance criteria:**
- [ ] `OllamaProvider` sends `tools` array in request body for capable models
- [ ] Tool call extracted from `message.tool_calls[0]` when present — `parseResponse()` not called
- [ ] Falls back to `parseResponse()` when `tool_calls` is absent or empty
- [ ] Existing 40 tests still pass (fallback path unchanged)
- [ ] `ai check` reports whether the selected model supports native tool use
- [ ] Tests: ≥4 — native tool call extracted correctly, fallback fires when absent, malformed tool_calls handled, mixed content (thought + tool_call) parsed

**API contract:**
```typescript
// OllamaProvider sends:
{
  model: "qwen2.5-coder:7b",
  messages: [...],
  tools: [
    {
      type: "function",
      function: {
        name: "bash",
        description: "...",
        parameters: { type: "object", properties: { command: { type: "string" } }, required: ["command"] }
      }
    }
  ],
  stream: false
}

// Response when model uses tool:
{
  message: {
    role: "assistant",
    content: "",           // may contain reasoning text
    tool_calls: [
      { function: { name: "bash", arguments: { command: "ls" } } }
    ]
  }
}
```

**Implementation note:** Tool definitions in `ToolDefinition` already match the OpenAI/Ollama function schema. The translation layer is minimal — wrap each definition in `{ type: "function", function: definition }`.

### F-10 · User-defined skills (skill.md files)

**Motivation:** Power users want to encode reusable workflows, personas, and constraints without writing TypeScript. A skill is a markdown file that defines a named behaviour — a custom system prompt extension, a set of task-specific rules, or a canned workflow — invokable by name from the CLI. This is the prompt-layer equivalent of F-06 (tool plugins): tools extend what the agent can *do*, skills extend how it *thinks and behaves*.

**What a skill is:**  
A markdown file at `~/.config/ai-cli/skills/<name>.md` with a structured frontmatter header followed by a body of instructions.

```markdown
---
name: refactor
description: Refactor TypeScript code for readability and strict-mode compliance
args:
  - name: target
    description: File or directory to refactor
    required: true
---

You are refactoring TypeScript. Rules:
1. Preserve all existing behaviour — do not change logic
2. Apply strict null checks and remove all implicit `any`
3. Extract repeated logic into named functions
4. After each file change, run `tsc --noEmit` to verify types
5. Target: {{target}}
```

**Behaviour:**
- Skills are loaded from `~/.config/ai-cli/skills/` on startup
- Invoked with `ai run --skill <name> [args]` or the shorthand `ai run /<name> [args]`
- Frontmatter is parsed for `name`, `description`, and `args` definitions
- `args` are positional or named; values are interpolated into `{{arg_name}}` placeholders in the body
- The skill body is injected into the system prompt **after** the standard rules block, as an additional "## Active Skill" section
- Skills do not replace the system prompt — they extend it
- `ai skills` lists all installed skills with their name and description
- Invalid frontmatter (missing `name`) logs a warning and skips the file — never crashes

**Frontmatter spec:**
```yaml
name: string          # required — used as the invocation key
description: string   # required — shown in `ai skills` list
args:                 # optional
  - name: string      # arg name, used as {{name}} placeholder
    description: string
    required: boolean  # default false
```

**Acceptance criteria:**
- [ ] Skill file loaded from `~/.config/ai-cli/skills/` at startup
- [ ] `ai run --skill refactor --target src/tools/bash.ts` injects skill body into system prompt
- [ ] `{{target}}` placeholder replaced with `src/tools/bash.ts` before injection
- [ ] `ai run /refactor src/tools/bash.ts` shorthand works identically
- [ ] `ai skills` lists all loaded skills with name and description
- [ ] Missing required arg prints clear error before run starts — does not call the model
- [ ] Invalid frontmatter (missing `name`) skips file with warning, does not crash
- [ ] Skill body appears under `## Active Skill` in the system prompt, after the rules block
- [ ] Two skills can be composed: `ai run --skill refactor --skill strict-types ...` (both injected)
- [ ] Tests: ≥5 — load valid skill, placeholder substitution, missing required arg, invalid frontmatter skipped, `ai skills` output

**File layout:**
```
~/.config/ai-cli/
├── config.json
├── logs/
├── tools/          ← F-06 (code plugins)
└── skills/         ← F-10 (prompt plugins)
    ├── refactor.md
    ├── deploy.md
    └── explain.md
```

**New files:**
- `src/agent/SkillLoader.ts` — reads, parses frontmatter, validates, interpolates args
- `src/cli/commands/skills.ts` — `ai skills` command

**Modified files:**
- `src/agent/promptBuilder.ts` — `buildSystemPrompt()` accepts optional `Skill[]` parameter, appends skill bodies
- `src/cli/commands/run.ts` — adds `--skill` flag, passes loaded skills to `toAgentConfig()`
- `src/cli/commands/chat.ts` — same `--skill` flag support
- `src/types.ts` — adds `Skill` type and `AgentConfig.skills?: Skill[]`

### F-11 · Skill scaffolding — built-in `create_skill` tool + default skill

**Motivation:** Asking users to hand-write frontmatter YAML and markdown is friction. The agent should be able to create skills for you. A built-in `create_skill` tool + a shipped default `skill-builder` skill lets users describe what they want in plain English and have the agent generate, validate, and save a skill file in one run.

**`create_skill` tool:**
- Writes a validated skill file to `~/.config/ai-cli/skills/<name>.md`
- Validates frontmatter before writing: `name` must be kebab-case, `args` must have names, no duplicate skill names
- Returns the written file path and a preview of the skill on success
- Returns a clear error if the name conflicts with an existing skill (with `overwrite: boolean` arg to force)

```typescript
definition.name: "create_skill"
parameters.required: ["name", "description", "body"]
parameters.optional: ["args", "tools", "overwrite"]
// args: Array<{ name: string; description: string; required?: boolean }>
// tools: string[] — permitted tool names (default: all)
// overwrite: boolean — replace existing skill (default: false)
```

**Shipped default skill — `skill-builder`:**
- Stored at `src/skills/skill-builder.md` and copied to `~/.config/ai-cli/skills/skill-builder.md` on first run
- Invoked with `ai run /skill-builder` or `ai run --skill skill-builder`
- Guides the agent to interview the user (via the task description), produce a skill definition, and call `create_skill` to save it
- Uses only `create_skill` and `bash` (to list existing skills) — no filesystem write access beyond the skills directory

**Skill body for `skill-builder`:**
```markdown
---
name: skill-builder
description: Interactively create a new ItsAAgent skill from a plain-English description
tools: [bash, create_skill]
---
You are helping the user create a new ItsAAgent skill file.

1. Read the user's task description to understand what they want to automate
2. Run `bash` with `ls ~/.config/ai-cli/skills/` to see existing skills
3. Design the skill: name (kebab-case), description, any args with {{placeholders}}, tool access list
4. Use the create_skill tool to save it
5. Confirm the skill was created and show the user how to invoke it
```

**Acceptance criteria:**
- [ ] `create_skill` tool writes a valid skill file to the skills directory
- [ ] Duplicate name without `overwrite: true` returns an error
- [ ] `skill-builder.md` is shipped with the project in `src/skills/`
- [ ] `skill-builder` is copied to the user skills dir on first run if not already present
- [ ] `ai run /skill-builder "create a skill that summarises git diffs"` produces a working skill file
- [ ] Tests: ≥4 — write skill, duplicate name error, overwrite flag, invalid frontmatter rejected

### F-12 · Per-skill tool access control

**Motivation:** A skill that fetches URLs shouldn't have access to `ssh`. A skill that only reads files shouldn't be able to run `bash`. Scoping tool access per skill reduces blast radius, makes skills more predictable, and lets users share skills without worrying about unintended side-effects.

**Behaviour:**
- Skill frontmatter accepts a `tools` key: a list of permitted tool names
- Default when `tools` is omitted: **all built-in tools permitted**
- When a skill with a `tools` list is active, `AgentRuntime` filters `this.tools` to only the permitted set
- The system prompt's tool descriptions section only lists permitted tools — the model cannot call what it cannot see
- If multiple skills are composed and they have different `tools` lists, the **union** of permitted tools is used
- `ai skills` displays the tool access list per skill
- Unknown tool names in `tools` list log a warning and are ignored — not a fatal error

**Frontmatter example:**
```yaml
---
name: safe-reader
description: Read and summarise files — no writes, no shell, no network
tools: [read_file, glob, grep]
---
```

**Acceptance criteria:**
- [ ] Skill with `tools: [read_file]` — agent cannot call `bash` or `ssh`
- [ ] Tool not in permitted list returns `"Tool not permitted by active skill"` error without executing
- [ ] Omitting `tools` key permits all tools
- [ ] Union of two composed skill tool lists is used when both are active
- [ ] System prompt only describes permitted tools when a skill is active
- [ ] `ai skills` shows permitted tools per skill
- [ ] Unknown tool name in `tools` list logs warning, does not crash
- [ ] Tests: ≥4 — permitted tool runs, blocked tool returns error, union of two skills, all-tools default

### F-13 · Enhanced interactive TUI — dashboard and configuration menus

**Motivation:** Running `ai` with no arguments does nothing. The CLI has no discoverable entry point for new users. Provider config requires manual JSON editing or flag memorisation. There is no way to browse installed tools or skills interactively. F-13 adds a full-screen interactive dashboard as the default `ai` experience.

**`ai` with no arguments — main menu:**
```
  ItsAAgent  ·  qwen2.5-coder:7b  ·  Ollama

  ❯ Run a task
    Chat
    ─────────────
    Tools
    Skills
    ─────────────
    Provider settings
    ─────────────
    History
    Quit
```

**Menu flows:**

**Run a task** → Clack `text()` prompt for task → `ai run` flow  
**Chat** → enters `ai chat` session  

**Tools** → full-screen list of all loaded tools (built-in + plugins):
```
  bash          Execute shell commands
  read_file     Read a file
  write_file    Write a file
  glob          Find files by pattern
  grep          Search file contents
  ssh           Run commands on remote servers
  ── plugins ──
  my_tool       My custom plugin tool
```
Select a tool → shows full description + parameter list

**Skills** → list of all installed skills:
```
  skill-builder   Interactively create a new skill   [bash, create_skill]
  refactor        Refactor TypeScript                 [all tools]
  safe-reader     Read and summarise files            [read_file, glob, grep]
```
Select a skill → preview body, option to run it or delete it

**Provider settings** → submenu:
```
  ❯ Type         ollama
    Model        qwen2.5-coder:7b
    Host / URL   http://localhost:11434
    API key      (not set)
    ─────────────
    Connect custom provider
    Test connection
    ← Back
```
- Each item is editable via Clack `text()` or `select()`
- Changes written to `~/.config/ai-cli/config.json` immediately
- **Connect custom provider** wizard:
  1. Select type: `ollama` / `openai-compat` / `anthropic` (future)
  2. Enter base URL
  3. Enter API key (optional, stored in config — warn user it's plain text)
  4. Select from available models (calls `listModels()` against the URL)
  5. Save and set as active provider
  6. Run `test connection` automatically

**History** → `ai history` list (F-07), select to view full log

**Acceptance criteria:**
- [ ] `ai` with no arguments renders the main menu
- [ ] All menu items navigate to their respective flows
- [ ] Provider settings edits are persisted to config.json
- [ ] Custom provider wizard validates URL before calling `listModels()`
- [ ] Invalid URL in custom provider shows error and returns to wizard (does not crash)
- [ ] Model list in provider wizard is fetched live from the entered URL
- [ ] Skills menu shows tool access list per skill
- [ ] Tools menu shows full parameter descriptions
- [ ] Menu is non-TTY safe — falls back gracefully if not in a terminal

**New files:**
- `src/cli/tui/Dashboard.tsx` — main menu Ink component
- `src/cli/tui/ProviderWizard.tsx` — provider setup flow
- `src/cli/tui/ToolsMenu.tsx` — tools browser
- `src/cli/tui/SkillsMenu.tsx` — skills browser

**Modified files:**
- `src/index.ts` — `ai` with no args renders `Dashboard` instead of showing help

## Implementation order

Features should be implemented in dependency order:

1. **F-09** (native tool use) — highest architectural impact; once merged, F-08 becomes maintenance rather than a primary fix
2. **F-08** (stronger parser) — now the fallback path; still needed for non-tool-use-capable models
3. **F-01** (edit_file) — self-contained tool
4. **F-02** (fetch) — self-contained tool
5. **F-03** (git) — self-contained tool
6. **F-05** (context eviction warning) — touches ContextManager and AgentRuntime
7. **F-04** (plain mode streaming / --raw) — touches CLI layer
8. **F-07** (ai history) — new command, touches SessionLogger
9. **F-10** (user-defined skills) — new `SkillLoader`, touches promptBuilder and CLI commands
10. **F-12** (per-skill tool access) — depends on F-10; filters tool registry per active skill
11. **F-11** (skill scaffolding) — depends on F-10 and F-12; `create_skill` tool + shipped default skill
12. **F-13** (interactive dashboard) — depends on F-07, F-10, F-06; pulls all menus together
13. **F-06** (plugin system) — touches tool registry and ai check

Each feature ships on its own `feat/` branch, merged to `main` individually. Do **not** batch features into one branch.
