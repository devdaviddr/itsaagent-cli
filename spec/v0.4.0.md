# ItsAAgent v0.4.0 — Feature Spec

**Branch:** `docs/v0.4.0-spec`
**Status:** Implemented (all 7 features, F-01…F-07) — released as v0.4.0
**Target:** v0.4.0 release (**minor** — additive, backward-compatible. The persistent TUI becomes the default *interactive* view; every existing CLI/config contract is preserved)
**Process:** Spec Driven Development — implementation begins only after the spec is approved. Tests are written against this spec.

---

## Vision

v0.3.0 made the agent **navigable** by bolting pickers and slash-commands onto three different UIs. v0.4.0 makes it **coherent**: a single, persistent, opencode-style terminal app. One full-screen view with a scrollable message log, a fixed input box, a status line, themed and collapsible tool blocks, and slash commands typed inline. Holding a conversation, switching agent/model, and browsing tools all happen **inside the same view** — no mode switches, no exits, no visual whiplash.

This is the release where the TUI stops being "Ink for run, Clack for menu, `console.log` for chat" and becomes one designed surface — **without breaking a single existing command.**

---

## Why this is a minor (not major) release

A persistent TUI sounds like a breaking change, but the public contract is preserved. The rule: the new TUI is **additive and opt-in for scripted paths**; only the *interactive* default surface changes.

| Surface | v0.3.0 | v0.4.0 | Breaking? |
|---|---|---|---|
| `iaa run "task"` (TTY) | One-shot Ink render, exits | **Unchanged default**: one-shot, exits | No |
| `iaa run "task" -i` | — | **New opt-in flag**: opens the persistent TUI seeded with the task | No (additive flag) |
| `iaa run` non-TTY / piped | Plain `console.error` renderer | **Unchanged** | No |
| `iaa chat` | Clack `text()` loop | Same conversation semantics, rendered in the new TUI | No (behaviour-compatible) |
| `iaa` (no args, TTY) | Clack home menu | Persistent TUI replaces the menu | No — interactive menu is not a stable CLI contract |
| All other subcommands (`config`, `check`, `tools`, `models`, `agents`, `skills`) | — | **Untouched** | No |
| Config schema | — | Optional `theme` field added | No (additive, optional) |
| `AgentRuntime` API | — | Additive `cancel()` method | No (additive) |

No flag is removed, no output that a script consumes changes, no config key changes meaning. Hence **0.4.0**, per the repo's semver rule ("minor — new tools/commands/flags, backward-compatible").

## Background

Today there are **three visually and behaviourally inconsistent UIs**, plus real bugs:

1. **`iaa run` uses Ink** (`AgentView.tsx`) as a one-shot, fire-and-forget render: stream a task, print an answer, unmount, exit. No input, no history, no scrollback.
2. **`iaa chat` does not use Ink at all** (`chatSession.ts`). It is a `@clack/prompts` `text()` loop printing answers with `console.log`. Completely different look from run.
3. **The home menu** (`menu.ts`) is a third style — Clack select lists.

On top of the inconsistency, the analysis surfaced concrete defects to retire as part of this work:

| # | Defect | Location |
|---|---|---|
| D-1 | Dead hook calls `runtime.removeAllListeners()` — global listener wipe | `src/cli/tui/hooks/useAgentEvents.ts` |
| D-2 | `onError` resolves the same promise as `onDone` — callers can't tell success from failure | `src/cli/output.ts` |
| D-3 | Hardcoded truncation (120/300 chars), no terminal-width awareness | `StepView.tsx`, `AgentView.tsx` |
| D-4 | Unbounded `currentChunk` string growth across a long run | `AgentView.tsx:90` |
| D-5 | Duplicated event-wiring logic (inline in `AgentView` vs the dead hook) | both files |
| D-6 | Usage thresholds (60/80) duplicated across three files | `output.ts`, `contextBar.ts`, `AgentView.tsx` |
| D-7 | No in-TUI cancellation — only `Ctrl+C` kills the whole process | `AgentView.tsx` |

v0.4.0 fixes all seven as a by-product of the rebuild — they are encoded as acceptance criteria, not left as "nice to have".

## Stack decision: stay on Ink (with discipline)

The user is open to alternatives, so this was evaluated before committing:

| Option | Verdict |
|---|---|
| **Ink 5 + React** (current) | **Chosen.** Already a dependency; team knows it; declarative state maps cleanly onto the existing EventEmitter surface; `<Static>` + `useStdout` + `useInput` cover scrollback, full-screen, and key handling. React reconciliation is the right model for a log that grows. |
| `neo-blessed` / `blessed` | Rejected. Imperative widget tree, effectively unmaintained, would mean a second TUI paradigm in the repo and discarding all current Ink code. |
| `terminal-kit` | Rejected. Lower-level; we'd hand-roll layout/scrolling that Ink gives us. |
| Custom ANSI (opencode is Go/Bubbletea) | Rejected. opencode's *look* is reproducible in Ink; rewriting its *framework* in raw ANSI is not worth it for a Node/TS project. |

**Conclusion:** the inconsistency is not Ink's fault — it's that only one of three UIs uses it. The fix is to make Ink the *single* interactive surface, add the missing primitives (input box, scroll region, theme), and remove Clack/`console.log` from the interactive path. **New dependency:** `ink-text-input` (controlled input box) — small, official Ink ecosystem. No framework change.

`@clack/prompts` stays for the **non-interactive one-shot subcommands** (`iaa config`, `iaa check`, etc.); it is removed only from the `chat`/menu interactive path.

## Architecture changes from v0.3.0

| Area | v0.3.0 | v0.4.0 |
|---|---|---|
| Default `iaa` (no args, TTY) | Clack home menu | **Persistent TUI app** (`App.tsx`) |
| `iaa run "task"` (TTY) | One-shot Ink render, then exit | **Same default**; `-i/--interactive` opens the persistent TUI seeded with the task |
| `iaa chat` (TTY) | Clack `text()` loop + `console.log` | Alias into the same TUI |
| Input | Clack prompts | In-TUI fixed input box (`ink-text-input`) |
| History | None (run) / scrollback-less (chat) | Scrollable message log with keyboard paging |
| Tool rendering | 120-char truncated one-liner | Themed, collapsible block (expand/collapse per call) |
| Slash commands | Parsed in `chatSession` only | Parsed in the TUI input, with an autocomplete menu |
| Theme | Ad-hoc `color=` props | Central `theme.ts`, one palette, config-selectable |
| Non-TTY / piped | Plain `console.error` renderer | **Unchanged** — `output.ts` plain path stays exactly as-is |
| Cancellation | Ctrl+C kills process | `Esc` cancels the in-flight run, TUI stays open; Ctrl+C still quits |

**Non-goals / preserved:** the agent core (`AgentRuntime`, events, providers, tools) is **not** modified except for one additive method (`cancel()`, see F-07). The plain non-TTY renderer is preserved verbatim for pipes/CI. Config schema is unchanged except an additive optional `theme` field.

## Features

---

### F-01 · Persistent TUI shell (`App.tsx`)

**Motivation:** One surface to replace three. This is the frame everything else lives in.

**Behaviour:**
- A full-screen Ink app: **header** (title · agent · model · context bar) at top, **scrollable message log** in the middle, **fixed input box** at the bottom, **hint/status line** under it.
- Launched by: `iaa` with no args in a TTY; `iaa chat`; and `iaa run "task" -i` (seeds the first message). `iaa run "task"` **without** `-i` keeps the existing one-shot render-and-exit path untouched.
- Uses the alternate screen buffer so the user's scrollback is restored on exit.
- Survives multiple turns: submit a message → agent runs → result appended to the log → input box ready again. No remount per turn.
- Owns a single `AgentRuntime` for the session via `continueChat()` semantics (context preserved across turns, matching today's chat).
- `Ctrl+C` (at idle) or `/exit` quits, restoring the terminal.

**Acceptance criteria:**
- [ ] `iaa` (no args, TTY) launches the persistent TUI
- [ ] `iaa chat` launches the same TUI; `iaa run "task" -i` launches it seeded with `task`
- [ ] `iaa run "task"` (no `-i`) behaves exactly as v0.3.0 (one-shot, exits) — regression-guarded by a test
- [ ] Header, scrollable log, fixed input box, and hint line all render and stay positioned as the log grows
- [ ] Multiple turns run in one session without remounting; context is preserved across turns
- [ ] On exit the original terminal scrollback is restored (alternate screen buffer)
- [ ] Non-TTY / piped invocation still uses the plain renderer unchanged
- [ ] Tests: ≥3 — event stream → log-entry reducer; turn-submission appends without clearing prior turns; non-TTY and no-`-i` paths still select the legacy renderer

**Files changed:**
- `src/cli/tui/App.tsx` — new root component (replaces `AgentView` as the interactive entry view)
- `src/cli/tui/layout/` — `Header.tsx`, `MessageLog.tsx`, `InputBox.tsx`, `StatusLine.tsx`
- `src/cli/output.ts` — TTY interactive path renders `App`; one-shot + plain paths untouched; fix D-2 (return `{ ok, answer }`)
- `src/cli/commands/run.ts` — add `-i/--interactive` flag; `commands/chat.ts`, `menu.ts` — route into `App`
- `src/index.ts` — no-args TTY launches `App`; register `-i` on `run`

### F-02 · Conversation model + scrollback

**Motivation:** opencode's core feel is a scrollable transcript you can page through.

**Behaviour:**
- The log is an ordered list of **entries**: `user`, `thought`, `tool` (collapsible — F-03), `answer`, `error`, `notice` (e.g. "context cleared").
- A reducer turns the AgentRuntime event stream into entries (replaces the ad-hoc `steps[]` patching in `AgentView`). This is the canonical state; D-5 (duplicated wiring) is resolved by having exactly one event→state mapping, unit-tested in isolation.
- Streaming `chunk` deltas render live into the in-progress `answer`/`thought` entry, then finalise — **bounded**: only the active entry buffers deltas; finalised entries hold the complete text, and the live buffer is not the whole-run concatenation (fixes D-4).
- **Keyboard scrollback:** `PgUp`/`PgDn` and arrow keys scroll the log while the input box stays fixed; `End`/`G` jumps to latest; auto-follows the tail unless the user has scrolled up.
- Terminal-width aware: entries wrap to `process.stdout.columns`, recomputed on `resize` (fixes D-3). No magic 120/300 constants.

**Acceptance criteria:**
- [ ] Event stream → entries reducer produces the correct ordered entry list for a representative run (thought → tool → tool result → answer)
- [ ] Streaming deltas render live then finalise; the live buffer is bounded to the active entry (no whole-run concatenation)
- [ ] `PgUp`/`PgDn` scroll history; `End` returns to the tail; new output auto-follows only when already at the tail
- [ ] Entries wrap to terminal width and reflow on terminal resize
- [ ] Tests: ≥3 — reducer over a recorded event sequence; bounded-buffer invariant; scroll-offset state machine (follow vs paused)

**Files changed:**
- `src/cli/tui/state/conversation.ts` — entry types + pure reducer (testable)
- `src/cli/tui/layout/MessageLog.tsx` — scroll region, width-aware wrapping
- `src/cli/tui/hooks/useAgentEvents.ts` — rewritten to feed the reducer; D-1 fixed (per-handler `off()`, never `removeAllListeners`)

### F-03 · Collapsible, themed tool blocks

**Motivation:** Replace the 120-char truncated one-liner with tidy, inspectable blocks.

**Behaviour:**
- Each tool call renders as a bordered block: header line `▸ <tool> <short args>  <status icon>`; body shows args and result.
- **Collapsed by default** once complete (header + 1-line summary); **expandable** to full args + full result. Toggle with `Enter`/`Space` on the focused block, or `Ctrl+R` to expand/collapse all.
- Status icon + colour come from the theme (F-04): running (spinner), success (✓), error (✗).
- Result rendering is type-aware where cheap: multi-line output keeps its newlines (no `.replace(/\n/g," ")`), long output is scroll-capped inside the block, not silently cut at 120 chars. If output is truncated for display, the block shows `… (N more lines — Enter to expand)` rather than hiding it.

**Acceptance criteria:**
- [ ] Completed tool calls render collapsed (header + summary) and expand to full args/result
- [ ] Expand/collapse works per-block and all-at-once; focus moves between blocks
- [ ] Multi-line results preserve newlines; over-long results show an explicit truncation marker with line count, never a silent cut
- [ ] Status icon/colour are theme-driven, consistent with the rest of the UI
- [ ] Tests: ≥2 — collapsed/expanded summary formatting helper; truncation-marker math (shown vs hidden line counts)

**Files changed:**
- `src/cli/tui/components/ToolBlock.tsx` — new (tool rendering for the persistent TUI)
- `src/cli/tui/state/conversation.ts` — `expanded` flag per tool entry
- `src/cli/tui/StepView.tsx` — **kept** for the legacy one-shot `AgentView` (the legacy
  renderer is intentionally retained, see "What's deferred"), so it is not removed.

### F-04 · Theming

**Motivation:** A coherent palette applied everywhere, configurable.

**Behaviour:**
- A single `theme.ts` exporting a `Theme` (semantic tokens: `user`, `assistant`, `thought`, `toolName`, `success`, `error`, `warning`, `muted`, `accent`, `border`, `ctxLow/Mid/High`). Every component reads tokens — **no raw `color="red"` literals** left in TUI components.
- Ships ≥2 built-in themes (e.g. `default` dark, `mono`). Selected via config `theme` field and `/theme <name>` in the TUI; falls back to `default` if unset/unknown.
- The context-usage thresholds (60/80) move into the theme/constants module — single source of truth (fixes D-6). `contextBar.ts` consumes them.

**Acceptance criteria:**
- [ ] All TUI components read colours from the theme; no raw colour literals remain in `src/cli/tui/`
- [ ] ≥2 built-in themes selectable via `config.theme` and `/theme <name>`; unknown/unset → `default`
- [ ] Context thresholds defined once and consumed by the bar and header
- [ ] `config.theme` is optional and backward-compatible (absent → default)
- [ ] Tests: ≥2 — theme resolver (name → palette, unknown → default); threshold constants are the single source used by `usageLevel`

**Files changed:**
- `src/cli/tui/theme.ts` — new
- `src/cli/contextBar.ts` — consume shared thresholds
- `src/cli/config.ts`, `src/types.ts` — optional `theme` field
- all `src/cli/tui/` components — swap literals for tokens

### F-05 · Inline slash commands + autocomplete

**Motivation:** opencode lets you drive everything from the input box.

**Behaviour:**
- Typing `/` in the input box opens an **autocomplete popup** above the box listing matching commands with one-line help.
- Commands (reuse the existing `parseChatInput` dispatcher from `chatCommands.ts`, extended): `/help`, `/agent <name>`, `/agents`, `/model <name>`, `/models`, `/theme <name>`, `/clear`, `/tools`, `/exit`.
- Executing a command appends a `notice` entry to the log (e.g. "Switched to plan — context cleared") instead of `console.error`. Unknown command → inline error entry, no crash.
- `/agent` and `/clear` reset context (as today) and post a notice; `/model` and `/theme` persist to config and update the header live.
- `/tools` opens an in-log browser of registered tools (replaces needing the separate Clack menu while in the TUI).

**Acceptance criteria:**
- [ ] `/` opens an autocomplete popup filtering as you type; Tab/Enter completes
- [ ] All listed commands work inside the TUI and post results as log entries (not `console.*`)
- [ ] `/agent`/`/clear` reset context with a notice; `/model`/`/theme` persist and update the header
- [ ] Unknown command yields an inline error entry, TUI stays alive
- [ ] Tests: ≥3 — autocomplete filter helper; command dispatch maps to actions; persistence side-effects for `/model` and `/theme`

**Files changed:**
- `src/cli/tui/components/CommandPalette.tsx` — autocomplete popup
- `src/cli/chatCommands.ts` — extend dispatcher (`/theme`, `/models`, `/tools`); keep pure/testable
- `src/cli/tui/App.tsx` — wire dispatch to conversation/notice entries

### F-06 · Status line + live header

**Motivation:** Always show state; replace the menu's separate status header.

**Behaviour:**
- **Header:** `ItsAAgent · <agent> · <model> · [ctx bar] NN%`. Updates live on agent/model/theme change and on `context:usage`.
- **Status/hint line** under the input: contextual hints — idle (`/help · Esc cancel · Ctrl+C quit`), running (`● running… Esc to cancel`), error state, scroll position when paged up.
- Provider-unreachable state shows `⚠ provider unreachable` in the header without breaking the TUI (mirrors v0.3.0 menu behaviour).

**Acceptance criteria:**
- [ ] Header shows agent, model, and a colour-coded context bar; updates live on change and on usage events
- [ ] Status line shows mode-appropriate hints (idle / running / scrolled / error)
- [ ] Unreachable provider shows a warning without breaking the TUI
- [ ] Tests: ≥2 — header formatting helper; status-line text selected by mode

**Files changed:**
- `src/cli/tui/layout/Header.tsx`, `src/cli/tui/layout/StatusLine.tsx`
- reuse `src/cli/contextBar.ts` helpers

### F-07 · In-TUI cancellation

**Motivation:** A stuck run shouldn't force killing the whole process (fixes D-7).

**Behaviour:**
- `AgentRuntime` gains an additive `cancel()` (cooperative: sets a flag checked in `runLoop` between steps / around the provider call; resolves the in-flight `run`/`continueChat` with a cancelled outcome). The agent core's existing event contract is otherwise unchanged.
- In the TUI, `Esc` while a run is in flight cancels it: the log gets a `notice` ("cancelled"), the input box returns, the session stays open.
- `Ctrl+C` at idle still quits; `Ctrl+C` during a run cancels then quits if pressed twice.

**Acceptance criteria:**
- [ ] `AgentRuntime.cancel()` stops the loop cooperatively and resolves the in-flight call as cancelled (no unhandled rejection)
- [ ] `Esc` during a run cancels and keeps the TUI open with a cancellation notice
- [ ] `Ctrl+C` at idle quits; double `Ctrl+C` during a run cancels then quits
- [ ] Non-cancelled runs behave exactly as before
- [ ] Tests: ≥2 — `cancel()` short-circuits the loop and emits/returns a cancelled result; idempotent if called when idle

**Files changed:**
- `src/agent/AgentRuntime.ts` — `cancel()` + cooperative check (additive)
- `src/cli/tui/App.tsx` — `useInput` Esc/Ctrl+C handling

## Implementation order (dependency-aware) — 4 branches

Feature IDs are grouped into four branches for a tighter sequence. Reducer and theme are pure modules built and unit-tested **before** the Ink layout, so most acceptance criteria are testable without rendering.

| # | Branch | Features | Rationale |
|---|---|---|---|
| 1 | `feat/tui-foundation` | **F-04** theme + **F-02** conversation reducer | Pure, fully unit-testable modules; no UI risk; everything else depends on them |
| 2 | `feat/tui-shell` | **F-01** shell + **F-06** header/status | The frame that hosts the reducer + layout regions and reads theme |
| 3 | `feat/tui-interaction` | **F-03** tool blocks + **F-05** slash commands | Interactive richness plugged into the live shell |
| 4 | `feat/tui-cancellation` | **F-07** cancellation | Touches the agent core; done last and in isolation |

## Branch strategy

```
main
 └── docs/v0.4.0-spec                 ← this branch (spec only)
      After approval:
      ├── feat/tui-foundation          (F-04, F-02)
      ├── feat/tui-shell               (F-01, F-06)
      ├── feat/tui-interaction         (F-03, F-05)
      └── feat/tui-cancellation        (F-07)
```

Each branch merges to `main` with `--no-ff` independently and must be build- and test-green before merge. `v0.4.0` is tagged (via a `chore/release-v0.4.0` branch) once all four are merged: bump `package.json` + `src/index.ts` `.version()`, roll `CHANGELOG.md` `[Unreleased]` → `[v0.4.0]`.

## Definition of done

A feature is done when:
- [ ] All acceptance criteria checked off
- [ ] `pnpm build` passes with zero errors
- [ ] `pnpm test` passes (all pre-existing + new tests); criteria reflected 1:1 in test descriptions
- [ ] Defects D-1…D-7 verified fixed (each maps to a criterion above)
- [ ] Backward-compat guard green: `iaa run "task"` (no `-i`, and piped) behaves as v0.3.0
- [ ] `CHANGELOG.md` updated under `[Unreleased]` (Added / Changed)
- [ ] `README.md` updated (the interactive experience changed) and `CLAUDE.md` architecture section updated for the new `src/cli/tui/` layout
- [ ] No `console.log` / `console.error` left in the interactive TUI path (plain renderer excepted)
- [ ] Smoke-tested on the real binary (`pnpm add -g .`, then `iaa`, `iaa run`, `iaa run -i`, `iaa chat`) including a piped/non-TTY run
