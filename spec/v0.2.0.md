# ItsAAgent v0.2.0 — Feature Spec

**Branch:** `feat/v0.2.0-spec`  
**Status:** Draft  
**Target:** v0.2.0 release  
**Process:** Spec Driven Development — implementation begins only after spec is approved. Tests are written against this spec, not against the implementation.

---

## Vision

v0.1.0 proved the core loop works. v0.2.0 makes the agent reliable and complete for real coding and infrastructure tasks: fix the broken chat experience, remove the fragile text parser, close the tool gaps that block dev workflow, and lay the groundwork for purpose-built agents.

No TUI rewrite in this release. The Ink UI ships as-is. Substance over chrome.

---

## Background

v0.1.0 shipped with one generic agent, Ink/React TUI, and text-based XML tool parsing. The post-release review identified these gaps in priority order:

1. **Broken chat memory** — `ai chat` wipes context on every message. The model cannot recall anything from the previous turn. This is a bug, not a feature gap.
2. **Tool parsing fragility** — the 3-fallback XML parser breaks on tool output containing XML-like strings. Ollama has supported native function calling since late 2024.
3. **Context explosion on large files** — `read_file` has no size guard. Reading a 3,000-line file blows the 24k context window; the ContextManager silently evicts earlier messages including potentially the original task.
4. **Silent context eviction** — when messages are trimmed, the model gets no notification. It continues as if history is complete, producing incoherent output.
5. **No surgical file edits** — `write_file` rewrites entire files. Large files fail silently; git diffs become unreadable.
6. **No git awareness** — the agent cannot propose commits, check diffs, or work in branches.
7. **Loop detection too narrow** — only catches exact-match repeats. Semantic loops (same intent, different args) run to max steps.
8. **No SSH file transfer** — the agent can run commands on remote servers but cannot push or pull files.

---

## Architecture changes from v0.1.0

| Layer | v0.1.0 | v0.2.0 |
|---|---|---|
| Chat memory | Wiped every message | Persistent across turns within a session |
| Tool calling | 3-fallback XML parser | Native Ollama function calling (XML as fallback) |
| `read_file` | No size guard, no line ranges | Size check + line range params |
| Context eviction | Silent | In-context notification to model |
| Agent model | Single generic agent | `AgentRegistry` with Build, Plan, CLI built-ins + user-defined agents |
| Extensibility | None | User-defined agents and skills loaded from `~/.config/ai-cli/` |
| SSH | Commands only | Commands + file upload/download |

### C-01 · Chat session continuity

**Motivation:** `ai chat` creates one `AgentRuntime` and loops, which looks correct. But each iteration calls `runtime.run(task)`, which calls `this.ctx.clear()` at line 1 — wiping the entire conversation history including the system prompt. The model has no memory of anything said before the current message. This is a clear bug.

**Root cause** (`src/agent/AgentRuntime.ts`):
```typescript
async run(task: string): Promise<string> {
  this.ctx.clear()  // ← wipes everything, every call
  this.ctx.add({ role: "system", content: buildSystemPrompt(...) })
  this.ctx.add({ role: "user", content: task })
```

**Fix:**
- Add `continueChat(task: string): Promise<string>` to `AgentRuntime`
- `continueChat` does NOT call `ctx.clear()` — it appends the new user message to existing context
- The system prompt is set once (on first `run()` call or a new `initSession()` method)
- After the agent answers, the assistant turn is already in context (it's added mid-loop) — the next `continueChat` call just appends the next user message
- `ai chat` calls `run()` for the first message, `continueChat()` for all subsequent messages
- `ai run` continues to use `run()` — single-task, stateless, unchanged

**Behaviour:**
```
> what is the capital of France?
Paris.

> what did i just ask you?
You asked me what the capital of France is.
```

**Acceptance criteria:**
- [ ] `ai chat` second message can reference content from the first response
- [ ] `ai chat` maintains context for the duration of the session (until `/exit`)
- [ ] Context trimming still works within a chat session (long sessions evict old turns, not the system prompt)
- [ ] `ai run` is unaffected — still clears context per invocation
- [ ] `/clear` command in chat resets context (calls `ctx.clear()` + re-inits system prompt)
- [ ] Long chat sessions trim oldest non-system turns first — the system prompt is never evicted
- [ ] Tests: ≥4 — multi-turn recall, context preserved after tool call, `/clear` resets, trim evicts oldest turn not system prompt

**Files changed:**
- `src/agent/AgentRuntime.ts` — add `continueChat()`, `initSession()`
- `src/cli/commands/chat.ts` — use `continueChat()` after first message

### F-09 · Native Ollama tool-use

**Motivation:** The 3-fallback XML parser is the single biggest reliability gap. Tool output containing XML-like strings (HTML, config files, log output) causes false tag matches. Ollama's native function calling makes structure the API's job, not ours. The existing `ToolDefinition` interface is already OpenAI-compatible — the translation is minimal.

**What changes:**
- `OllamaProvider` sends `tools: [...]` in the request body for capable models
- Capability detected at startup via `listModels()` — models with `tools` in capabilities use native calling; others fall back to text parsing
- `AgentRuntime` reads `message.tool_calls[0]` directly — `parseResponse()` not called
- `parseResponse()` becomes the fallback path for non-capable models only
- `thought` still extracted from `message.content` when present alongside tool calls
- **Context turn in native mode:** the assistant turn added to `ctx` is `message.content` (the reasoning text, even if empty string) — not the raw HTTP response body. This keeps the two paths consistent: both add a text assistant turn to context before executing the tool call.

**Tool definition translation** (minimal — already compatible):
```typescript
// From existing ToolDefinition:
{ name: "bash", description: "...", parameters: { ... } }

// To Ollama format:
{ type: "function", function: { name: "bash", description: "...", parameters: { ... } } }
```

**Acceptance criteria:**
- [ ] `tools` array sent in request body for capable models
- [ ] `tool_calls[0]` extracted without calling `parseResponse()`
- [ ] Falls back to `parseResponse()` when `tool_calls` absent or model not capable
- [ ] All 40 existing tests pass (fallback path unchanged)
- [ ] `ai check` reports: `✓ native tool use` or `✗ text parsing fallback` for the active model
- [ ] Tests: ≥4 — native extraction, fallback fires, malformed tool_calls handled, mixed content (thought + tool_call)

**Files changed:**
- `src/providers/OllamaProvider.ts` — send tools, detect capability
- `src/agent/AgentRuntime.ts` — read tool_calls before parseResponse

### R-01 · `read_file` line ranges and size guard

**Motivation:** `read_file` has no upper bound. Reading a 5,000-line file consumes ~14k tokens — over half the context window — in a single tool call. The ContextManager then silently evicts earlier messages (including potentially the task itself) with no indication to the model that this happened. The model continues with incomplete history.

**Changes to `read_file`:**
- Add optional `start_line` (1-indexed, inclusive) and `end_line` (1-indexed, inclusive) parameters
- Before reading: stat the file. If size > 150 KB, return:
  ```
  { success: false, error: "File is 340 KB — too large to read whole (limit: 150 KB).
    Use start_line/end_line to read a section, or grep to search it." }
  ```
- If `start_line`/`end_line` provided: read only those lines, return content with a header:
  ```
  [Lines 45–90 of 312 in src/tools/ssh.ts]
  <content>
  ```
- If no range and file is within limit: read whole file (existing behaviour)
- Out-of-range line numbers return a clear error (don't clamp silently)

**System prompt addition** (rule 9 — see R-05):
```
9. Before reading any file of unknown size, run bash: wc -l <path> first.
   If it exceeds 300 lines, use read_file with start_line/end_line (both 1-indexed, inclusive).
```

**Acceptance criteria:**
- [x] `read_file` with `start_line`/`end_line` returns only those lines with range header
- [x] File > 150 KB without range returns error with guidance message
- [x] File within limit with no range: reads whole file (unchanged behaviour)
- [x] Out-of-range line numbers return error
- [x] Tests: ≥5 — whole small file, range read, oversized rejection, out-of-range, range header format

**Files changed:**
- `src/tools/filesystem.ts` — extend `read_file` tool definition and execute

### R-02 · Context eviction notification

**Motivation:** `ContextManager.trim()` splices messages silently. The model sees a gap in its history with no explanation — tool results disappear, earlier reasoning is gone. It continues as if nothing changed, producing incoherent follow-up thoughts.

**Fix:** When `trim()` removes one or more messages, inject a synthetic user message immediately after the eviction point:

```
[CONTEXT NOTICE: 3 messages were trimmed to stay within the context window.
The original task and the most recent tool results are preserved.]
```

This message is not logged to the session file (it's a runtime artefact). It IS sent to the model on the next provider call.

**`ContextManager` changes:**
- `trim()` returns the count of evicted messages (currently `void`)
- `add()` checks the return value; if > 0, splices in the notice message at eviction position
- Notice message uses `role: "user"` so it appears in the conversation stream (some models ignore `system` mid-conversation)
- The notice message itself is never evicted (pinned like the system prompt and original task)

**Acceptance criteria:**
- [ ] Filling context past `maxTokens` causes the notice to appear in `ctx.get()`
- [ ] Notice is not written to the session log
- [ ] Notice message is never itself evicted
- [ ] `ctx.usage()` accurately reflects trimmed state
- [ ] Tests: ≥3 — notice injected on eviction, notice not logged, notice not evictable

**Files changed:**
- `src/agent/ContextManager.ts` — `trim()` returns count, `add()` injects notice

### F-01 · `edit_file` tool

**Motivation:** `write_file` rewrites entire files. This makes git diffs unreadable, fails silently on large files (model generates truncated content), and causes conflicts in multi-step edits where the model writes the file twice with different content.

**Behaviour:**
- Parameters: `path`, `start_line` (1-indexed, inclusive), `end_line` (1-indexed, inclusive), `new_content`
- Reads the file, replaces lines `start_line..end_line` with `new_content`, writes back
- Returns a unified diff of the change so the model can verify
- `end_line = start_line - 1` (end < start) → pure insertion before `start_line`, nothing removed
- `new_content` empty string → deletes the specified lines
- Missing file → `{ success: false, error: "No such file: <path>" }`
- Out-of-range → `{ success: false, error: "Line X out of range (file has N lines)" }`

**Acceptance criteria:**
- [ ] Replaces a range and returns a unified diff
- [ ] Inserts lines without removing content when `end < start`
- [ ] Empty `new_content` deletes the line range
- [ ] Fails gracefully on missing file
- [ ] Fails gracefully on out-of-range line numbers
- [ ] Does not rewrite the entire file
- [ ] Tests: ≥5 — replace, insert, delete, missing file, out-of-range

**Files changed:**
- `src/tools/filesystem.ts` — add `editFileTool`
- `src/tools/index.ts` — register tool

### F-03 · `git` tool

**Motivation:** The agent makes code changes but cannot propose commits, check what changed, or work in branches. This is the single largest workflow gap for coding tasks.

**Behaviour:**
- Parameter: `subcommand` (required), `args` (optional string), `cwd` (optional, defaults to `process.cwd()`)
- Executes via `execFile("git", [subcommand, ...args.split(" ")])` — no shell injection
- `commit` subcommand: validates that `args` contains `-m` before running — returns error if missing
- Output truncated at 6 KB

**Allowed subcommands:** `status`, `diff`, `log`, `add`, `commit`, `branch`, `checkout`, `show`, `stash`

**Blocked subcommands:** `push --force`, `reset --hard`, `clean`, `rebase` — return `{ success: false, error: "subcommand not permitted" }` without executing

**Acceptance criteria:**
- [ ] `git status` returns stdout
- [ ] `git add <file>` stages a file
- [ ] `git commit -m "msg"` creates a commit
- [ ] Blocked subcommands return error without executing
- [ ] `commit` without `-m` returns error before running git
- [ ] Tests: ≥5

**Files changed:**
- `src/tools/` — add `git.ts`
- `src/tools/index.ts` — register tool

### R-03 · SSH file transfer (`ssh_upload` / `ssh_download`)

**Motivation:** The agent can run commands on remote servers (including Wake-on-LAN recovery, connection multiplexing, sudo retry) but cannot move files. Pushing a config file, pulling a log, or deploying a binary all require file transfer. This is the gap that makes the SSH story incomplete.

**Two new tools:**

**`ssh_upload`** — copies a local file to a remote path:
```typescript
parameters.required: ["host", "user", "local_path", "remote_path"]
parameters.optional: ["port", "key_path"]
// Executes: scp -P <port> -i <key> <local_path> <user>@<host>:<remote_path>
// Uses same key/password auth pattern as existing ssh tool
// Returns: "Uploaded <local_path> → <user>@<host>:<remote_path> (N bytes)"
```

**`ssh_download`** — copies a remote file to a local path:
```typescript
parameters.required: ["host", "user", "remote_path", "local_path"]
parameters.optional: ["port", "key_path"]
// Executes: scp -P <port> -i <key> <user>@<host>:<remote_path> <local_path>
// Returns: "Downloaded <user>@<host>:<remote_path> → <local_path> (N bytes)"
```

Both tools:
- Use `safeEnv()` (strips `SSH_PASS`) consistent with the existing SSH tool
- Timeout after 60s (file transfers can be slow)
- Return clear errors for auth failures, missing files, unreachable hosts
- **Password auth caveat:** `scp` password auth requires `sshpass` to be installed on the system. If `sshpass` is not available and no key path is provided, return `{ success: false, error: "Password auth for scp requires sshpass (brew install sshpass). Use key auth instead: set key_path." }` — do not silently hang waiting for a password prompt.

**Acceptance criteria:**
- [ ] `ssh_upload` uploads a local file to remote host
- [ ] `ssh_download` downloads a remote file to local path
- [ ] Auth failure returns clear error
- [ ] Missing local file (upload) returns error before attempting scp
- [ ] Timeout fires at 60s
- [ ] Password auth without `sshpass` returns a clear error, does not hang
- [ ] Tests: ≥4 (mock execFile — same pattern as ssh tool tests; include sshpass-missing path)

**Files changed:**
- `src/tools/ssh.ts` — add `sshUploadTool`, `sshDownloadTool`
- `src/tools/index.ts` — register both

### R-04 · Smarter loop and failure recovery

**Motivation:** The current loop detector catches only exact repeats (`stableKey` — same tool, same args, 3× in a row). Semantic loops run to `maxSteps`: the model reads `file_a.ts`, then `file_b.ts`, then `file_a.ts` again in a cycle. Tool failure loops also run to exhaustion: the model gets an error, tweaks the command slightly, gets the same error class, repeats.

**Two improvements:**

**a) Recency-window loop detection:**
- Track the last 8 tool calls in a sliding window (name only, not args)
- If the same tool name appears 5+ times in the window, inject a context nudge as a user message:
  ```
  [AGENT NOTICE: You have called read_file 5 times recently.
  Do you have enough information to proceed, or are you stuck?
  State what you're looking for before calling another tool.]
  ```
- This is a nudge, not a hard abort — the model can still continue if it reasons about it

**b) Failure escalation:**
- Per-tool failure counter (separate from the loop counter)
- After 2 consecutive failures with the same tool name, inject:
  ```
  [AGENT NOTICE: bash has failed twice in a row. Before trying again,
  state explicitly why the previous attempts failed and what you will do differently.]
  ```
- After 3 consecutive failures: hard abort with `{ error: "Tool <name> failed 3 times consecutively" }`
- The per-tool failure counter resets to 0 on any successful call to that tool

**Acceptance criteria:**
- [ ] Recency nudge fires when same tool used 5+ times in last 8 calls
- [ ] Nudge is a user message in context, not a hard abort
- [ ] Failure escalation injects warning after 2 consecutive same-tool failures
- [ ] Hard abort fires after 3 consecutive same-tool failures
- [ ] Per-tool failure counter resets to 0 on a successful call to that tool
- [ ] Existing exact-match loop detection (`stableKey`) still works alongside new detection
- [ ] Tests: ≥5 — recency nudge triggers, failure warning triggers, hard abort triggers, counter resets on success, exact-match still works

**Files changed:**
- `src/agent/AgentRuntime.ts` — failure counter, recency window, nudge injection

### A-01 · Agent registry and built-in agents

**Motivation:** One generic agent with full tool access gives the model too much surface area and too much blast radius. A read-only analysis agent cannot accidentally write files. Scoped tool access also reduces system prompt length — the model only sees tools it can use.

**Three built-in agents (CLI only — no TUI picker in this release):**

| Agent | Purpose | Tool access | Flag |
|---|---|---|---|
| `build` | Full-access development work — edit, run, commit | all tools | `--agent build` (default) |
| `plan` | Read-only analysis — no mutations, no shell | `read_file`, `glob`, `grep`, `git`, `fetch` | `--agent plan` |
| `cli` | Shell and infrastructure — commands, SSH, file transfer | `bash`, `ssh`, `ssh_upload`, `ssh_download`, `fetch`, `download_file` | `--agent cli` |

**`AgentDefinition` interface:**
```typescript
interface AgentDefinition {
  id: string
  name: string
  description: string
  tools: string[] | "all"
  readonly: boolean             // blocks all mutation tools regardless of `tools`
  model?: string                // optional model override
  systemPromptSuffix?: string   // appended after standard rules block
}
```

**`AgentRegistry`:**
- Instantiated at startup; holds built-in agents
- `AgentRuntime` receives resolved `AgentDefinition`, filters tool set, adjusts system prompt
- `--agent <id>` CLI flag on `ai run` and `ai chat`
- `ai agents` command lists all registered agents

**Acceptance criteria:**
- [x] `build` agent has access to all tools
- [x] `plan` agent cannot call `bash`, `write_file`, `edit_file`, `append_file`, `delete_file`, `ssh`, `ssh_upload`, `ssh_download` — returns `"Tool not permitted by active agent"` without executing
- [x] `cli` agent cannot call `read_file`, `write_file`, `edit_file`, `append_file`, `delete_file`, `glob`, `grep`, `git`
- [x] System prompt only describes permitted tools for the active agent
- [x] Default agent is `build`
- [x] `--agent plan` and `--agent cli` flags select the respective agents
- [x] `ai agents` lists all three agents with description and tool count
- [x] Tests: ≥5 — tool permitted, tool blocked (plan), tool blocked (cli), system prompt scoped, default selection

**Files changed:**
- `src/agent/AgentDefinition.ts` — new interface + built-in definitions
- `src/agent/AgentRegistry.ts` — new registry class
- `src/agent/AgentRuntime.ts` — accept `AgentDefinition`, filter tools
- `src/cli/commands/run.ts` — `--agent` flag
- `src/cli/commands/chat.ts` — `--agent` flag
- `src/cli/commands/agents.ts` — new `ai agents` command
- `src/index.ts` — register `agents` command

### U-01 · Context usage indicator

**Motivation:** In `ai chat`, the user has no visibility into how full the context window is. They don't know when messages are about to be evicted, can't tell why the agent seems to forget things, and have no way to make an informed decision about starting a new session. The data already exists (`ContextManager.usage()` returns `{ total, max, ratio }`) — it just needs to be surfaced.

**What it looks like:**

In the TUI header (existing `AgentView.tsx`), add a second line beneath the model/cwd line:

```
ItsAAgent  ·  qwen2.5-coder:7b  ·  ~/projects/myapp

ctx  [████████░░░░░░░░]  8,432 / 24,576  34%
```

Colour coding on the bar and percentage:
- **< 60%** — dim/default (no concern)
- **60–80%** — yellow (approaching limit)
- **> 80%** — red (eviction likely soon)

In plain/non-TTY mode and after each `ai chat` response: print to stderr when context ratio ≥ 60%, always when ratio ≥ 80%:
```
[ctx: 8,432/24,576 — 34%]
```

**How it works:**

- `AgentRuntime` emits a new `context:usage` event after every `ctx.add()` call:
  ```typescript
  "context:usage": [payload: { used: number; max: number; ratio: number }]
  ```
- `AgentView.tsx` subscribes to `context:usage` and updates a state field `contextUsage`
- The header renders the bar and values from `contextUsage`
- Plain mode output handler (`output.ts`) subscribes to `context:usage` and conditionally prints to stderr
- The bar is rendered with plain ASCII chars (`█` filled, `░` empty), width 16 characters

**`context:usage` firing points:**
1. After the system prompt is added (`ctx.add` in `run()`)
2. After each user message
3. After each assistant turn
4. After each tool result
This means the display updates in real time as the run progresses.

**Acceptance criteria:**
- [ ] `context:usage` event emitted after every `ctx.add()` call with correct `{ used, max, ratio }` values
- [ ] TUI header shows bar, token counts, and percentage
- [ ] Bar is dim/default below 60%, yellow 60–80%, red above 80%
- [ ] Plain mode prints ctx line to stderr when ratio ≥ 60%
- [ ] Plain mode always prints ctx line after `ai chat` completes a response (regardless of ratio)
- [ ] `ai run` also shows context usage in verbose mode (`-v`)
- [ ] Tests: ≥3 — event fires with correct values, ratio colour thresholds, plain mode prints at correct threshold

**Files changed:**
- `src/agent/AgentRuntime.ts` — add `context:usage` event type + emit after every `ctx.add()`
- `src/cli/tui/AgentView.tsx` — subscribe to event, render bar in header
- `src/cli/output.ts` — subscribe to event in plain mode, print at threshold

### A-02 · User-defined agents

**Motivation:** The three built-in agents cover common patterns but every developer's workflow is different. A `reviewer` that only reads and comments, a `devops` agent scoped to infrastructure tools, a `writer` with no code tools — these are real patterns that shouldn't require modifying source. Agents are markdown files with YAML frontmatter, loaded at startup alongside the built-ins.

**Format** (`~/.config/ai-cli/agents/<name>.md`):
```markdown
---
name: reviewer
description: Review code changes for correctness and clarity — read-only, no edits
tools: [read_file, glob, grep, git]
readonly: true
model: qwen2.5-coder-7b-32k:latest
---

You are a code reviewer. Your job is to find bugs, unclear logic, and missing edge cases.
Focus on correctness first, clarity second. Do not suggest style changes unless they
affect readability in a meaningful way.
After reading each file, summarise your findings before moving to the next.
```

**Frontmatter fields:**
```yaml
name: string         # required — used as the --agent key and shown in ai agents
description: string  # required — shown in ai agents list
tools: string[]      # optional — list of permitted tool names; omit to allow all tools
readonly: boolean    # optional — if true, all mutation tools blocked regardless of tools list
model: string        # optional — overrides the configured model for this agent
```

**Behaviour:**
- Loaded from `~/.config/ai-cli/agents/` at startup alongside built-in agents
- The agent body (everything after frontmatter) is appended to the system prompt under `## Agent Instructions`
- Invalid frontmatter (missing `name`): logs warning to stderr and skips — never crashes
- Name collision with a built-in (`build`, `plan`, `cli`) is an error: `"Agent name 'build' is reserved"` — skip the file
- `ai agents` shows built-ins first, then user agents separated by a blank line, tagged `[custom]`
- Invoked with `--agent <name>` identically to built-ins

**File layout:**
```
~/.config/ai-cli/
├── config.json
├── logs/
├── agents/          ← A-02 (user-defined agents)
│   └── reviewer.md
└── skills/          ← X-02 (user-defined skills)
    └── refactor.md
```

**Acceptance criteria:**
- [ ] Valid agent file loads and is selectable via `--agent <name>`
- [ ] Agent body injected under `## Agent Instructions` in system prompt
- [ ] `tools` list respected — unlisted tools return `"Tool not permitted"` without executing
- [ ] `readonly: true` blocks all mutation tools regardless of `tools` list
- [ ] `model` override applied when frontmatter sets it
- [ ] Name collision with built-in skips file with clear warning
- [ ] Invalid frontmatter skips with warning, does not crash
- [ ] `ai agents` shows `[custom]` tag for user agents, built-ins first
- [ ] Tests: ≥5 — load valid agent, tool restriction, readonly flag, model override, name collision rejected

**Files changed:**
- `src/agent/AgentRegistry.ts` — scan `~/.config/ai-cli/agents/`, parse frontmatter, merge with built-ins
- `src/agent/AgentLoader.ts` — new: frontmatter parsing and validation for agent files

### T-01 · `delete_file` tool

**Motivation:** The model currently deletes files via `bash: rm -rf path`. This goes through a shell with no guardrails — the model could construct `rm -rf /` or accidentally glob-expand into unintended files. A dedicated tool enforces safe deletion without a shell.

**Behaviour:**
- Parameter: `path` (required) — path to a single file or empty directory
- Refuses wildcards: if `path` contains `*`, `?`, or `[` returns `{ success: false, error: "Wildcards not allowed. Use glob to find files, then delete individually." }`
- Refuses any path that resolves inside a `.git` directory
- Refuses non-empty directories: `{ success: false, error: "Directory is not empty. Delete contents first or use bash." }`
- On success, files: `"Deleted <path> (N bytes)"`
- On success, empty dirs: `"Deleted directory <path>"`
- Uses `fs.unlink` / `fs.rmdir` — no shell

**Acceptance criteria:**
- [ ] Deletes a file and returns size confirmation
- [ ] Refuses wildcard paths with guidance message
- [ ] Refuses paths inside `.git/`
- [ ] Refuses non-empty directories
- [ ] Missing file returns clear error (does not crash)
- [ ] Tests: ≥5 — file delete, wildcard rejection, .git rejection, non-empty dir rejection, missing file

**Files changed:**
- `src/tools/filesystem.ts` — add `deleteFileTool`
- `src/tools/index.ts` — register tool

### T-02 · `download_file` tool

**Motivation:** `fetch` truncates responses at 8 KB — it is for reading and inspecting content, not downloading files. The agent cannot retrieve a binary, a large config file, a release archive, or any content over 8 KB without a dedicated download tool. `download_file` writes the response body directly to disk with no content limit.

**Behaviour:**
- Parameters: `url` (required), `destination` (required — local file path)
- HTTP and HTTPS only — other schemes rejected
- Follows redirects (max 5)
- Timeout: 120s (downloads can be slow; distinct from `fetch`'s 15s)
- Streams response to disk — does not load into memory
- Creates parent directories if needed
- Does not strip or transform content — raw bytes
- On success: `"Downloaded <url> → <destination> (N bytes, <content-type>)"`

**Acceptance criteria:**
- [ ] Downloads a URL and writes to destination path
- [ ] Non-HTTP(S) scheme rejected with clear error
- [ ] Parent directories created if missing
- [ ] 120s timeout fires
- [ ] Returns byte count and content-type on success
- [ ] Redirect limit (5) respected
- [ ] Tests: ≥4 — successful download, non-HTTP rejection, timeout, redirect limit

**Files changed:**
- `src/tools/filesystem.ts` — add `downloadFileTool`
- `src/tools/index.ts` — register tool

### T-03 · `append_file` tool

**Motivation:** `write_file` overwrites the entire file. To append a line to `.gitignore`, add an entry to a config file, or log output without a full rewrite, the model must `read_file` → construct full content → `write_file`. This risks losing file content if the model constructs it incorrectly, and is unnecessarily expensive on large files.

**Behaviour:**
- Parameters: `path` (required), `content` (required)
- Appends `content` to the end of the file exactly as provided — no automatic newline added
- Creates the file if it does not exist (same as `write_file`)
- Creates parent directories if needed
- Returns: `"Appended N bytes to <path> (file is now M bytes total)"`

**Acceptance criteria:**
- [ ] Appends to existing file without modifying prior content
- [ ] Creates file if it does not exist
- [ ] Parent directories created if missing
- [ ] Returns appended byte count and new total file size
- [ ] Tests: ≥4 — append to existing, create new, parent dir creation, byte counts accurate

**Files changed:**
- `src/tools/filesystem.ts` — add `appendFileTool`
- `src/tools/index.ts` — register tool

### X-02 · Skill system

**Motivation:** Skills let users encode reusable behaviours without writing TypeScript. A `refactor` skill that sets strict TypeScript rules, a `review` skill that focuses on correctness, a `deploy` skill that knows your deployment workflow — these are prompt-layer workflows that compose with any agent. Unlike agents (which define a whole persona and tool set), skills are additive: they change *how* the current agent thinks without changing *what* it can do.

**Format** (`~/.config/ai-cli/skills/<name>.md`):
```markdown
---
name: refactor
description: Refactor TypeScript for readability and strict-mode compliance
args:
  - name: target
    description: File or directory to refactor
    required: true
---

You are refactoring TypeScript. Apply these rules strictly:
1. Preserve all existing behaviour — do not change logic
2. Remove all implicit `any`; use `unknown` and narrow where needed
3. Apply strict null checks throughout
4. After each file change, run `tsc --noEmit` to verify types
5. Target: {{target}}
```

**Frontmatter fields:**
```yaml
name: string          # required
description: string   # required — shown in ai skills list
args:                  # optional
  - name: string
    description: string
    required: boolean  # default false
```

**Behaviour:**
- Loaded from `~/.config/ai-cli/skills/` at startup
- Invoked with `--skill <name>` or `/name` shorthand: `ai run /refactor src/`
- Skill body injected into system prompt under `## Active Skill`, after standard rules block
- `{{arg_name}}` placeholders replaced with provided values before injection
- Missing required arg: exits with clear error before any model call
- Invalid frontmatter (missing `name`): logs warning and skips — never crashes
- Multiple skills composable: `--skill refactor --skill strict-types` (both injected in order)
- `ai skills` lists all loaded skills with name and description
- Skills do **not** restrict tool access — that is the agent's responsibility

**Invocation examples:**
```bash
ai run --skill refactor --target src/tools/bash.ts "clean up this file"
ai run /refactor src/tools/bash.ts
ai chat --agent plan --skill reviewer
```

**Acceptance criteria:**
- [ ] Skill body injected under `## Active Skill` in system prompt
- [ ] `{{arg}}` placeholders replaced before injection
- [ ] Missing required arg exits with clear error before model call
- [ ] Invalid frontmatter skips with warning, does not crash
- [ ] `/name` shorthand works identically to `--skill name`
- [ ] Multiple `--skill` flags compose — both bodies injected in order
- [ ] `ai skills` lists all skills with name and description
- [ ] Skill does not alter the active agent's tool permissions
- [ ] Tests: ≥5 — body injection, placeholder substitution, missing required arg, invalid frontmatter skipped, multi-skill composition

**Files changed:**
- `src/agent/SkillLoader.ts` — new: scan dir, parse frontmatter, validate, interpolate args
- `src/agent/promptBuilder.ts` — `buildSystemPrompt()` accepts optional `Skill[]`, appends bodies
- `src/cli/commands/run.ts` — `--skill` flag + `/name` shorthand parsing
- `src/cli/commands/chat.ts` — `--skill` flag
- `src/cli/commands/skills.ts` — new `ai skills` command
- `src/index.ts` — register `skills` command
- `src/types.ts` — add `Skill` type, extend `AgentConfig` with `skills?: Skill[]`

### CLI-02 · Interactive home menu (`iaa` with no arguments)

**Motivation:** Running `iaa` with no arguments currently shows the commander help block — a wall of text that tells you nothing about what the tool can actually do. A guided menu is the right default: it discovers what's available (models, agents, skills), surfaces the most common actions, and gives new users a path in without reading docs.

**Implementation:** Uses `@clack/prompts` (already a dependency) — no new packages, no TUI framework. Clack's `select()` handles arrow-key navigation, styling, and keyboard handling out of the box.

**What it looks like:**

```
  ┌  ItsAAgent
  │
  ◇  qwen2.5-coder:7b  ·  Ollama  ·  http://localhost:11434
  │
  ◆  What would you like to do?
  │  ❯ Run a task
  │    Chat
  │    ──────────────
  │    Agents          3 loaded
  │    Skills          2 loaded
  │    ──────────────
  │    Settings
  │    Quit
  └
```

**Menu flows:**

**Run a task** → Clack `text({ message: "Task:" })` → runs the full `iaa run` flow inline (same `AgentRuntime`, same TUI output). Esc/cancel returns to menu.

**Chat** → enters `iaa chat` session directly.

**Agents** → prints the `ai agents` list (name, description, tool count, `[custom]` tag), then returns to menu with a `press Enter to go back` prompt.

**Skills** → same pattern with `ai skills` list.

**Settings** → sequential Clack prompts for: provider type, host URL, API key, model. Reads current values from config as defaults. Writes on confirmation. Same flow as current `iaa config` but presented in the menu context.

**Quit** → clean exit.

**Context line** (`◇` line below the title) fetches the active model and provider from config synchronously — no Ollama call. If Ollama is unreachable when the menu opens, the line shows `⚠  Ollama unreachable` in amber but the menu still renders normally. The user can still navigate to Settings to fix the host.

**Acceptance criteria:**
- [ ] `iaa` with no arguments renders the menu (not commander help)
- [ ] Arrow keys navigate, Enter selects, Esc/Ctrl+C exits cleanly
- [ ] Context line shows active model and provider from config (no network call)
- [ ] Context line shows `⚠  Ollama unreachable` if host is down, menu still works
- [ ] **Run a task**: prompts for task text, runs agent, returns output, then re-renders menu
- [ ] **Chat**: enters chat session; returning from chat re-renders menu
- [ ] **Agents**: prints agent list, returns to menu on Enter
- [ ] **Skills**: prints skill list, returns to menu on Enter
- [ ] **Settings**: sequential prompts for all config fields, writes on confirm, returns to menu
- [ ] **Quit**: exits with code 0
- [ ] Non-TTY (piped output): skips menu, shows commander help as before
- [ ] Tests: ≥3 — non-TTY falls back to help, menu renders in TTY, settings writes config

**Files changed:**
- `src/cli/menu.ts` — new: `showHomeMenu()` function using Clack `select()`, `text()`, `outro()`
- `src/index.ts` — when `process.argv.length === 2` and `process.stdout.isTTY`, call `showHomeMenu()` instead of `program.help()`
