## Vision

A chat becomes a **session**: a unit that owns its own context. Sessions can be driven by an **advised process** ŌĆö a reusable, multi-stage workflow where the agent plans, **asks the user to clarify ambiguities**, and, once confident, **hands the work off to the build agent with a compact summary of what it learned**. The first process is `guided` (plan ŌåÆ clarify ŌåÆ confirm ŌåÆ build), but the primitives are universal: any agent can ask the user a question, any session can change agent, and the process layer is data-driven so more processes can be added later.

---

## Background (what's wrong today)

`AgentRuntime` conflates four concerns ŌĆö **context**, **active agent**, **model/provider**, and the **execution loop** ŌĆö and there is no first-class session:

1. **Context is trapped in the runtime and wiped on agent switch.** `setAgent()` + the next `run()`/`initSession()` clears `ContextManager`.
2. **The planŌåÆbuild handoff is lossy.** It re-seeds build with only the plan *text*; everything plan learned (files read, tool results) is discarded.
3. **No clarification primitive.** The ReAct loop runs to `<answer>`/maxSteps; the agent cannot pause to ask the user and resume.
4. **The planŌåÆbuild flow is hardcoded in `App.tsx`** ŌĆö no reusable process abstraction.
5. **Two context models** ŌĆö the runtime's `ContextManager` and the TUI's display reducer ŌĆö with no single source of truth.
6. **Heavy prompts** ŌĆö 14 numbered rules + verbose suffixes.

---

## Design principles

- **Session owns context.** The runtime becomes an engine operating on a session.
- **Primitives over features.** `ask_user` and context-preserving agent-switch are general; `guided` is built *from* them.
- **Compact, deterministic handoff.** No extra LLM call: the handoff seed = the plan + an auto-extracted list of what plan examined.
- **Minimal prompts, measured.** Slim the rules; re-measure tool-call reliability against the existing "create a file"/"build express api" checks.
- **Backward compatible.** Plain chat and `iaa run` keep working; sessions/processes are additive.

## Features

### F-01 ┬Ę `Session` ŌĆö first-class context owner

A `Session` owns: `id`, `title`, `model`, `cwd`, a `ContextManager`, the **active agent**, a structured **tool history** (`{ name, args }[]`), and an agent-transition log. `AgentRuntime` is refactored to **operate on a session** (state moves out of the runtime into the session; `run`/`continueChat`/`setAgent`/`setModel` delegate to it). Display (the TUI reducer) stays separate, but the **session is the single source of truth for model context**.

**Acceptance criteria:**
- [ ] `Session` holds context + active agent + model + cwd + meta + tool history; `AgentRuntime` reads its loop state from the session.
- [ ] Multi-turn chat preserves context within a session (parity with today's `continueChat`).
- [ ] `session.setAgent(def)` switches the agent and records the transition **without** clearing context (context handling on handoff is F-02).
- [ ] Tool calls append to the session's tool history.
- [ ] Tests: Ōēź4 ŌĆö session add/get/usage; agent switch keeps messages; tool history records `{name,args}`; model switch.

**Files:** new `src/agent/Session.ts`; refactor `src/agent/AgentRuntime.ts`; minor `src/cli/tui/launch.ts`/`App.tsx` (construct a session).

---

### F-02 ┬Ę Context handoff = plan + compact summary

`session.handoffTo(buildAgent, planText)`:
1. Build a **deterministic compact summary** from the session's tool history ŌĆö files read (`read_file`/`grep`/`glob` paths), commands run (`bash`), and edits made ŌĆö no extra LLM call.
2. Reset the model context to: `system`(build prompt) + one seed `user` message = the plan **plus** the "Planning context" summary.
3. Switch active agent to build and continue.

> Build inherits the plan and *what plan examined*, not the raw read-dumps (per the chosen "plan + compact summary" model).

**Acceptance criteria:**
- [ ] Handoff seeds build with the plan text + an auto-summary listing files examined and commands run during planning.
- [ ] Raw planning tool-result dumps are **not** carried into build's context (token-lean).
- [ ] After handoff the active agent is build and its tools/prompt are in scope.
- [ ] `plan` remains read-only up to the handoff (user-gated, F-04).
- [ ] Tests: Ōēź3 ŌĆö summary extraction from a tool history (files + commands); handoff produces a build system prompt + seed; empty-history handoff still carries the plan.

**Files:** `src/agent/Session.ts` (`handoffTo`, `examinedSummary`); `src/cli/tui/App.tsx` (use it in `handoffToBuild`).

---

### F-03 ┬Ę `ask_user` ŌĆö clarification primitive (universal)

A built-in pseudo-tool available to **every** agent (always permitted, including read-only `plan` ŌĆö asking isn't a mutation). When the model calls `ask_user(question)`, the runtime invokes an `onAskUser(question): Promise<string>` handler, **awaits the user's answer**, and returns it as the `[TOOL RESULT]` ŌĆö the loop resumes with the same context. The runtime emits an `ask` event; the TUI shows the question and captures the next input as the answer (a "pending question" input mode).

**Acceptance criteria:**
- [ ] `ask_user` appears in every agent's tool set + system prompt; calling it pauses the loop until the handler resolves, then resumes with the answer as the tool result.
- [ ] Read-only `plan` may call `ask_user` (not blocked).
- [ ] TUI: a pending question switches the input to "answer" mode; the next submission resolves it (not a new turn); Esc cancels the run.
- [ ] Without an `onAskUser` handler (e.g. non-TTY `iaa run`), `ask_user` returns a clear "no interactive user available" result so the loop doesn't hang.
- [ ] Tests: Ōēź3 ŌĆö `ask_user` resolves via the handler and feeds the result back; permitted under a read-only agent; no-handler fallback.

**Files:** `src/agent/AgentRuntime.ts` (handler + special-case in `executeTool`/`toolSpecs`); `src/agent/Session.ts`; `src/cli/tui/App.tsx` (pending-question mode); `src/agent/promptBuilder.ts` (one-line usage note).

---

### F-04 ┬Ę `guided` process (built-in)

A small, **data-driven** process: ordered stages, each binding an agent, advanced by a **user-confirmed** transition. The built-in `guided` = `[plan, build]`:

- Start `guided <task>` ŌåÆ session begins in `plan`, seeded with the task.
- `plan` analyses, uses `ask_user` for ambiguities, and ends with a concrete plan; the status line shows `Tab ŌåÆ hand off to build` (exists).
- On **Tab** ŌåÆ `session.handoffTo(build, plan)` (F-02) ŌåÆ `build` executes.

Entry points: a `/guided <task>` slash command (and the no-arg start hint). For v0.6.0 the process set is **hardcoded** (one built-in); the shape is built to accept a markdown loader later (deferred).

**Acceptance criteria:**
- [ ] A `Process` type describes ordered stages; `guided` is registered as the built-in.
- [ ] `/guided <task>` starts the session in plan seeded with the task; Tab advances planŌåÆbuild via the F-02 handoff.
- [ ] The advised flow works for **any** task (not express-specific) ŌĆö verified with two unrelated tasks.
- [ ] Tests: Ōēź2 ŌĆö process stage advancement (pure state machine); `guided` definition shape.

**Files:** new `src/agent/Process.ts` (+ built-in registry); `src/cli/chatCommands.ts` (`/guided`); `src/cli/tui/App.tsx` (drive stages).

---

### F-05 ┬Ę Slim prompts (measured)

Consolidate the 14 rules to ~7 sharp, high-impact ones; push tool specifics into tool descriptions (already there); keep agent suffixes short; add a one-line `ask_user` note and a tiny process-framing line. **Re-measure** tool-call reliability with the existing checks (`create a file`, `build express api`) on `qwen2.5-coder-7b-32k:latest` and `:7b`; the slim prompt must be **Ōēź** the current reliability.

**Acceptance criteria:**
- [ ] System prompt rule count reduced (~14 ŌåÆ ~7) with no loss in the kept high-value rules (anti-hallucination, one-tool-per-response, tool-call format, cwd-persists/OS-appropriate, ask_user).
- [ ] Measured reliability on the standard checks is **not worse** than v0.5.x (recorded in the spec/CHANGELOG).
- [ ] Tests: Ōēź1 ŌĆö prompt still contains the kept critical rules.

**Files:** `src/agent/promptBuilder.ts`, `src/agent/AgentDefinition.ts`.
