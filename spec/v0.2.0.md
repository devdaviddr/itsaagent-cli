# ItsAAgent v0.2.0 — Feature Spec

**Branch:** `feat/v0.2.0-spec`  
**Status:** Draft (revised — supersedes previous draft)  
**Target:** v0.2.0 release  
**Process:** Spec Driven Development — implementation begins only after spec is approved. Tests are written against this spec, not against the implementation.

---

## Vision

v0.1.0 proved the core loop works. v0.2.0 makes it feel like a real tool: a multi-agent system with purpose-built agents, a proper full-screen TUI, and a clean extensibility story for tools, skills, and custom agents.

The design is inspired by OpenCode's philosophy — specialised agents, keyboard-driven menus, great terminal UX — but built our own way: markdown-first agent definitions, a modal rather than panel menu system, and OpenTUI's SolidJS renderer instead of Ink/React for genuinely responsive terminal rendering.

---

## Background

v0.1.0 shipped with one generic agent, Ink/React TUI, and text-based XML tool parsing. The review identified four gaps:

1. **Single agent** — one system prompt with full tool access is the wrong default. A read-only analysis agent and a shell/infra agent serve different tasks better.
2. **TUI architecture** — Ink re-renders the full React tree on every keypress. OpenTUI's SolidJS fine-grained reactivity only updates cells that changed. Menus and real-time streaming both feel smoother.
3. **Tool parsing fragility** — the 3-fallback XML parser breaks on tool output that contains XML-like strings. Ollama has supported native function calling since late 2024; using it makes structure the API's job, not ours.
4. **No extensibility surface** — users cannot add tools, skills, or agents without modifying source.

---

## Architecture changes from v0.1.0

| Layer | v0.1.0 | v0.2.0 |
|---|---|---|
| TUI renderer | Ink (React) | OpenTUI + SolidJS |
| Agent model | Single `AgentRuntime` | `AgentRegistry` + typed `AgentDefinition` |
| Tool calling | 3-fallback XML parser | Native Ollama function calling (XML as fallback) |
| Extensibility | None | Tools, skills, agents all user-loadable |

**OpenTUI on Node.js:** confirmed working — ships platform-specific native binaries as optional npm deps (`@opentui/core-darwin-arm64` etc.), same pattern as esbuild. No Bun required.

### A-01 · Agent registry and built-in agents

**Motivation:** One generic agent with full tool access is a blunt instrument. A coding task needs write access and git; a research task should be read-only. Giving the model only the tools it needs for the job reduces noise in the system prompt and limits blast radius.

**Three built-in agents:**

| Agent | Purpose | Tool access | Default mode |
|---|---|---|---|
| `build` | Full-access development work — edit, run, commit | all tools | primary |
| `plan` | Read-only analysis and exploration — no mutations | `read_file`, `glob`, `grep`, `git` (status/diff/log only), `fetch` | primary |
| `cli` | Shell and infrastructure tasks — commands, SSH, scripting | `bash`, `ssh`, `fetch` | primary |

**`AgentDefinition` interface:**
```typescript
interface AgentDefinition {
  id: string                    // "build" | "plan" | "cli" | user-defined
  name: string                  // display name
  description: string           // shown in agent picker
  systemPrompt?: string         // overrides default if set
  tools: string[] | "all"       // permitted tool names, or "all"
  model?: string                // optional model override
  readonly: boolean             // if true, mutation tools are blocked regardless of `tools`
}
```

**`AgentRegistry`:**
- Instantiated at startup, holds built-in + user-loaded agents
- `AgentRuntime` receives the resolved `AgentDefinition` and filters its tool set accordingly
- Active agent stored in session config, switchable mid-session via TUI

**Acceptance criteria:**
- [ ] `build` agent has access to all tools
- [ ] `plan` agent cannot call `bash`, `write_file`, `edit_file`, `ssh` — returns `"Tool not permitted by active agent"` without executing
- [ ] `cli` agent cannot call `read_file`, `glob`, `grep`, `edit_file`, `write_file`
- [ ] `AgentRuntime` system prompt only lists permitted tools for the active agent
- [ ] Default agent on startup is `build`
- [ ] `--agent plan` CLI flag selects agent for the run
- [ ] Tests: ≥4 — tool permitted, tool blocked, system prompt contains only permitted tools, default agent selection

### A-02 · User-defined agents

**Motivation:** Power users want to define project-specific agents — a `reviewer` that only reads and comments, a `devops` agent with SSH and infrastructure tools, a `writer` with no code tools. The format mirrors skills: markdown with YAML frontmatter.

**Format** (`~/.config/ai-cli/agents/<name>.md`):
```markdown
---
name: reviewer
description: Review code changes — read-only, no edits
tools: [read_file, glob, grep, git]
model: qwen2.5-coder-7b-32k:latest
readonly: true
---

You are a code reviewer. Focus on correctness, clarity, and potential bugs.
Do not suggest style changes unless they affect readability significantly.
After reading each file, state your findings concisely before moving on.
```

**Behaviour:**
- Loaded at startup from `~/.config/ai-cli/agents/`
- Invalid frontmatter (missing `name`) logs a warning and skips — never crashes
- User agents appear in the agent picker alongside built-ins
- `ai agents` lists all loaded agents (built-in + user) with description and tool count

**Acceptance criteria:**
- [ ] Valid agent file loads and appears in picker
- [ ] Agent system prompt is injected after the standard rules block
- [ ] `tools` list respected — unlisted tools blocked
- [ ] `readonly: true` blocks all mutation tools regardless of `tools` list
- [ ] Invalid frontmatter skips with warning
- [ ] `ai agents` command lists all agents
- [ ] Tests: ≥4 — load valid agent, tool restriction, readonly flag, invalid frontmatter skipped

**File layout:**
```
~/.config/ai-cli/
├── config.json
├── logs/
├── tools/          ← X-01 (code plugins)
├── skills/         ← X-02 (prompt plugins)
└── agents/         ← A-02 (agent definitions)
```

### U-01 · OpenTUI + SolidJS TUI foundation

**Motivation:** Ink re-renders the full React tree on every event. OpenTUI uses SolidJS signals — only terminal cells that actually changed are redrawn. For a streaming agent that emits chunks every ~50ms and a menu system that responds to every keypress, the difference is noticeable. This is a complete replacement of `src/cli/tui/`.

**What changes:**
- Remove `ink`, `react`, `@types/react` dependencies
- Add `@opentui/core`, `@opentui/solid`, `@opentui/keymap`, `solid-js`
- Rewrite `AgentView.tsx` → `AgentView.tsx` (SolidJS component, same events, different renderer)
- Rewrite `StepView.tsx`, `Spinner.tsx` as SolidJS components
- Entry point (`renderTui()`) uses `render()` from `@opentui/solid` + `createCliRenderer()` from `@opentui/core`

**Component structure:**
```
src/cli/tui/
├── app.tsx              ← root SolidJS component, route switcher
├── context/
│   ├── agent.tsx        ← active agent signal
│   ├── session.tsx      ← session state
│   └── theme.tsx        ← colours / styles
├── views/
│   ├── SessionView.tsx  ← main conversation view (replaces AgentView)
│   ├── StepView.tsx     ← single step (thought/tool/result)
│   └── HomeView.tsx     ← launch screen (no active session)
├── dialogs/
│   ├── ModelPicker.tsx  ← U-03
│   ├── AgentPicker.tsx  ← U-04
│   └── SettingsDialog.tsx ← U-05
└── components/
    ├── Spinner.tsx
    ├── Header.tsx       ← agent · model · step count
    └── Footer.tsx       ← input bar + keybind hints
```

**Layout:**
```
┌─ ItsAAgent ──────────────── build · qwen2.5-coder:7b · step 3/25 ─┐
│                                                                      │
│  ✓ [bash] ls -la src/                                               │
│  ✓ [read_file] src/tools/bash.ts                                    │
│  ● [bash] tsc --noEmit          ← active step (spinner)            │
│    > checking types...                                               │
│                                                                      │
│                                                                      │
├──────────────────────────────────────────────────────────────────────┤
│  ^A agent  ^M model  ^H history  ^C cancel  ^Q quit                 │
└──────────────────────────────────────────────────────────────────────┘
```

**Acceptance criteria:**
- [ ] `npm run build` passes with OpenTUI deps and no Ink/React deps
- [ ] Streaming chunks render without full-screen flicker
- [ ] Step list scrolls when it exceeds terminal height
- [ ] Non-TTY fallback: renders plain text when not in a terminal (existing `renderPlain` path kept)
- [ ] All existing agent events (`start`, `step`, `chunk`, `thought`, `tool:call`, `tool:result`, `answer`, `error`) wired correctly
- [ ] Tests: existing integration tests pass; new snapshot test for step rendering

### U-02 · Modal keyboard menu system

**Motivation:** OpenCode uses Tab to switch agents and keyboard shortcuts for menus. We use a modal approach — a keybind opens a floating picker overlay, Esc closes it, Enter confirms. No persistent panels or sidebars; the session view is always full-screen.

**Keybindings:**

| Key | Action |
|---|---|
| `Ctrl+A` | Open agent picker |
| `Ctrl+M` | Open model picker |
| `Ctrl+,` | Open settings |
| `Ctrl+H` | Open session history |
| `Ctrl+C` | Cancel current run |
| `Ctrl+Q` or `q` | Quit |
| `↑ ↓` | Navigate within any picker |
| `Enter` | Confirm selection |
| `Esc` | Close any open dialog |
| `/` | Open command palette (future) |

**Pickers are overlays**, not routes — rendered on top of the session view with a dimmed background. State remains visible underneath.

**Acceptance criteria:**
- [ ] Each keybind opens the correct dialog
- [ ] Esc always closes the current dialog without changing state
- [ ] Selection takes effect immediately (model/agent switch applies to next run step)
- [ ] Only one dialog open at a time — opening a second closes the first
- [ ] Tests: keyboard event → dialog open/close state transitions

### U-03 · Model picker (Ctrl+M)

**Behaviour:**
- Lists all models available from the configured Ollama host (calls `OllamaProvider.listModels()`)
- Shows model name + parameter size if detectable from name
- Current model highlighted
- Fuzzy search: type to filter (same session, no re-fetch)
- Selection updates `config.model` and persists to `config.json`
- Shows `⚡ native tool use` badge for models with `tools` in capabilities

```
┌─ Model ─────────────────────────────┐
│  > _                                │
│  ─────────────────────────────────  │
│  ● qwen2.5-coder:7b   ⚡ native    │

│    llama3.2:3b        ⚡ native    │
│    codellama:13b                     │
│                                      │
│  ↑↓ navigate  Enter select  Esc back│
└─────────────────────────────────────┘
```

**Acceptance criteria:**
- [ ] Lists models from Ollama
- [ ] Fuzzy filter narrows list on keystroke
- [ ] Selection persists to config.json
- [ ] `⚡ native` badge shown for capable models
- [ ] Error shown if Ollama unreachable (does not crash)

### F-09 · Native Ollama tool-use

*(Carried from previous draft — architecture priority)*

**Motivation:** The 3-fallback XML parser is the biggest reliability gap. Ollama's native function calling (OpenAI-compatible) makes the API layer enforce structure. The existing `ToolDefinition` schema is already OpenAI-compatible — the translation is minimal.

**What changes:**
- `OllamaProvider` sends `tools: [...]` in request body for capable models
- Runtime reads `message.tool_calls[0]` directly — no `parseResponse()` call
- `parseResponse()` becomes the fallback for non-capable models only
- Model capability detected at startup via `listModels()` capabilities field

**Acceptance criteria:**
- [ ] Tools sent in request body for capable models
- [ ] `tool_calls` extracted without calling `parseResponse()`
- [ ] Fallback fires when `tool_calls` absent
- [ ] Existing 40 tests still pass
- [ ] `ai check` reports native tool use support for the selected model
- [ ] Tests: ≥4

### X-02 · Skill system

*(Carried from previous draft — renamed from F-10, scope trimmed)*

**Format** (`~/.config/ai-cli/skills/<name>.md`):
```markdown
---
name: refactor
description: Refactor TypeScript for readability and strict-mode compliance
args:
  - name: target
    description: File or directory to refactor
    required: true
---

You are refactoring TypeScript. Rules:
1. Preserve all existing behaviour
2. Apply strict null checks, remove implicit `any`
3. After each file change, run `tsc --noEmit`
Target: {{target}}
```

**Behaviour:**
- Loaded from `~/.config/ai-cli/skills/`
- Invoked with `--skill <name>` or `/<name>` shorthand
- Body injected into system prompt under `## Active Skill`
- `{{arg}}` placeholders replaced before injection
- `ai skills` lists all skills
- Multiple skills composable: `--skill a --skill b`
- Per-skill `tools` key supported (filters available tools for that skill)

**Acceptance criteria:**
- [ ] Skill loads and injects into system prompt
- [ ] Placeholder substitution works
- [ ] Missing required arg errors before calling the model
- [ ] `ai skills` lists all skills
- [ ] `tools` key restricts available tools
- [ ] Tests: ≥5

## Implementation order

Ordered by dependency and risk — riskiest last.

1. **A-01** — Agent registry + built-ins (pure TypeScript, no TUI, unblocks everything)
2. **F-01** — `edit_file` (self-contained, high value for build agent)
3. **F-02** — `fetch` (self-contained)
4. **F-03** — `git` (self-contained)
5. **F-09** — Native Ollama tool-use (architecture fix, unblocks reliable tool calling)
6. **A-02** — User-defined agents (depends on A-01)
7. **X-01** — Tool plugin system (depends on agent registry)
8. **X-02** — Skill system (depends on prompt builder)
9. **U-01** — OpenTUI foundation (riskiest — full TUI rewrite, do after core is stable)
10. **U-02** — Keyboard menu system (depends on U-01)
11. **U-03** — Model picker (depends on U-01, U-02)
12. **U-04** — Agent picker (depends on U-01, U-02, A-01)
13. **U-05** — Provider settings (depends on U-01, U-02)
14. **U-06** — Session history browser (depends on U-01, U-02)

## Definition of done

A feature is done when:
- [ ] All acceptance criteria checked off
- [ ] `npm run build` passes with zero errors
- [ ] `npm test` passes (all pre-existing tests + new feature tests)
- [ ] Spec acceptance criteria reflected 1:1 in test descriptions
- [ ] `CHANGELOG.md` updated under `[Unreleased]`
- [ ] No `console.log` left in production paths
