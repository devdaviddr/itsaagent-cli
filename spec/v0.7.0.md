# Spec v0.7.0 — Toward a highly dynamic local-model agent harness

## Vision
Turn `iaa` from a well-engineered *static* ReAct loop into a *dynamic* harness that
plans, gathers what it needs, verifies its own work, and adapts to the model in
front of it — reliably, on small local models. Roadmap derived from a 7-dimension
multi-agent review (loop-control, prompting, tools, context/memory, agents,
providers, testing).

## Background (the three gaps the review found)
1. **Completion is trusted on faith** — the loop returns the first `<answer>` with
   no check that the work happened; no verify step; the e2e harness scores only
   pass/fail.
2. **Everything is frozen at startup** — prompt never adapts to model/task/native
   mode; plan→build needs a human Tab; sampling is hardcoded for all models.
3. **Silent downstream truncation** — `maxContextTokens` is trimmed client-side but
   never sent to Ollama as `num_ctx`; `git` runs in `process.cwd()` not the session
   cwd. Well-built client abstractions defeated downstream.

## Phases (implementation order)

### Phase 0 — Correctness (do first, alone)
- **F-0.1 num_ctx**: thread `maxContextTokens` into `ProviderConfig` and send it to
  Ollama as `options.num_ctx`. *AC:* provider request includes `num_ctx`; config
  carries it. Files: `types.ts`, `cli/config.ts`, `providers/OllamaProvider.ts`.

### Phase 1 — Deterministic quick wins (no LLM cost)
- **F-1.1 SUCCESS/FAILED tool framing**: `formatToolResult` leads with `— OK`/`— FAILED`
  and appends a fix-or-rethink line on failure. *AC:* failed results say FAILED.
- **F-1.2 git session cwd**: `git` uses `getSessionCwd()`/`resolveSessionPath()`.
  *AC:* `cd` then `git status` targets the right repo (test).
- **F-1.3 mid-task `<answer>` re-prompt**: a pure `looksLikeMidTaskAnswer()` heuristic;
  on a build run, re-prompt once instead of accepting a status-shaped answer.
  *AC:* capped once per run; plan answers unaffected; unit-tested heuristic.
- **F-1.4 few-shot exemplar**: one worked trajectory in the prompt, behind a
  `fewShot` flag (default on) so the e2e harness can A/B it. *AC:* present when on,
  absent when off (test).

### Phase 2 — Make dynamism measurable
- **F-2.1 trajectory scoring**: e2e harness records steps-to-answer, tool count,
  tool errors, repeated calls, recovery-nudge fired (events already emitted).
- **F-2.2 baseline + `--compare`**: commit `tests/e2e/baseline.json`; diff pass-rate
  and avg steps; non-zero exit on regression beyond threshold.

### Phase 3 — Enforce completion (core self-correction)
- **F-3.1 run_tests/verify tool**: auto-detect npm/pytest/cargo/make; normalized
  PASS/FAIL + first failing lines, in the session cwd.
- **F-3.2 verification gate**: before accepting a build `<answer>` for a run that ran
  mutation tools, inject one `[VERIFY]` turn (capped once per run).
- **F-3.3 best-effort recovery**: replace hard 3-failure/max-steps aborts with one
  recovery turn (switch strategy / ask_user / summarize partial).

### Phase 4 — Per-model prompt dynamism
- **F-4.1 native-mode prompt branch**: pass tool-use mode into `buildSystemPrompt`;
  drop the XML `<tool_call>` schema when the model uses native function calls.

### Phase 5 — Structural orchestration
- **F-5.1 ProcessRunner**: auto-advance plan→build via `handoffToBuild` when a stage
  emits a non-empty `<answer>`; expose `iaa run --process guided` + headless guided.

### Phase 6 — Provider depth
- **F-6.1 constrained decoding**: Ollama `format` (JSON schema) for tool calls when
  native mode is on; text parser stays as fallback.
- **F-6.2 per-model profiles**: `modelProfiles.ts` keyed by name pattern (temperature,
  num_predict, num_ctx, stop, preferToolFormat); replace hardcoded 0.15/8192.
- **F-6.3 retries/backoff**: bounded retries on network/5xx/empty completions.

## Definition of done (per phase)
- `pnpm build` + `pnpm test` green; new ACs reflected in tests.
- Reliability not worse on the e2e harness (after Phase 2, measured via `--compare`).
- CHANGELOG `[Unreleased]` updated. No version bump until the owner asks.

## Top risk
Every verify/re-prompt/recovery intervention MUST be capped to fire **once per run**,
or premature-stopping becomes never-stopping on small models.
