## F-01 · ReAct agent loop

**What was built:**  
A Thought → Action → Observation loop running inside `AgentRuntime`. The model reasons in `<thought>` blocks, calls one tool per step via `<tool_call>`, receives the result as a `[TOOL RESULT]` user message, and repeats until it produces an `<answer>` tag or the step limit is reached.

**Acceptance criteria:**
- ✅ Model sees its own reasoning chain across all steps (assistant turn appended before tool result)
- ✅ `<answer>` tag terminates the loop and returns the answer text
- ✅ Unstructured text with no tags is treated as a final answer (`isExplicitAnswer: false`)
- ✅ Thought-only response (no tool call, no answer) triggers a reprompt: `"Continue. Use a tool or provide your <answer>."`
- ✅ Loop terminates at `maxSteps` with a `MaxStepsError` message

**Files:** `src/agent/AgentRuntime.ts`

## F-02 · Three-fallback response parser

**What was built:**  
`parseResponse()` handles the varied output formats of local 7b models. Tries three strategies in order before falling back to treating the whole response as a final answer.

**Fallback chain:**
1. `<tool_call>...</tool_call>` XML block (primary)
2. Legacy `TOOL: name {args}` line format (backward compat)
3. Bare JSON `{"name":"...","args":{...}}` scoped to text after `</thought>` (guards against false positives from JSON in reasoning text)

**Acceptance criteria:**
- ✅ XML `<tool_call>` block parsed correctly
- ✅ Legacy `TOOL:` format parsed correctly
- ✅ Bare JSON after `</thought>` parsed as tool call
- ✅ JSON embedded inside `<thought>` content is NOT parsed as a tool call
- ✅ `isExplicitAnswer: true` only when `<answer>` tag is present
- ✅ Malformed JSON in `<tool_call>` does not throw — falls through to next fallback
- ✅ Missing `args` in `<tool_call>` defaults to `{}`
- ✅ `stableKey()` produces order-independent loop detection keys

**Tests:** `tests/agent/parser.test.ts` — 10 tests  
**Files:** `src/agent/parser.ts`

## F-03 · Loop detection

**What was built:**  
Same tool + same args called 3 or more times aborts the run with `LoopDetectedError`. Key is computed with `stableKey()` which sorts JSON keys before stringifying — prevents the same call with reordered args from bypassing detection.

**Acceptance criteria:**
- ✅ Same tool + args repeated 3× emits `error` event with code `LOOP_DETECTED`
- ✅ `run()` returns the error message string (does not throw)
- ✅ Key comparison is order-independent

**Tests:** `tests/agent/runtime.test.ts`  
**Files:** `src/agent/AgentRuntime.ts`, `src/agent/errors.ts`

## F-04 · Context management

**What was built:**  
`ContextManager` is an append-only message store with token estimation (3.5 chars/token). When total tokens exceed `maxTokens`, it evicts oldest messages — but always preserves index 0 (system prompt) and index 1 (original user task).

**Acceptance criteria:**
- ✅ System prompt (index 0) is never evicted
- ✅ Original task (index 1) is never evicted
- ✅ Oldest non-pinned messages are evicted first
- ✅ Newest tool result survives over older ones when trimming
- ✅ `forProvider()` strips internal timestamps before sending to the model
- ✅ `usage()` returns `{ total, max, ratio }`
- ✅ `clear()` keeps only the system message

**Tests:** `tests/agent/context.test.ts` — 7 tests  
**Files:** `src/agent/ContextManager.ts`

## F-05 · Provider abstraction

**What was built:**  
A `Provider` interface with `stream()`, `checkHealth()`, and `listModels()`. Two implementations: `OllamaProvider` (NDJSON streaming) and `OpenAICompatProvider` (SSE streaming). `createProvider()` factory selects based on `ProviderConfig.type`.

**Acceptance criteria:**
- ✅ `OllamaProvider.stream()` yields `StreamChunk` objects from NDJSON
- ✅ `OllamaProvider.checkHealth()` hits `/api/tags` and checks `res.ok` (not just reachability)
- ✅ `OpenAICompatProvider` reads API key from `AI_API_KEY` or `OPENAI_API_KEY` env only — never from config file
- ✅ `:latest` suffix in model name normalised when checking availability

**Files:** `src/providers/Provider.ts`, `src/providers/OllamaProvider.ts`, `src/providers/OpenAICompatProvider.ts`, `src/providers/index.ts`

## F-06 · Six built-in tools

**What was built:**  
All tools implement the `Tool` interface. `execute()` never throws — always returns `ToolResult`. Args are defensively coerced. Tools are stateless.

### bash
- Executes via `execFile("bash", ["-c", command])` — no shell injection
- 30 s timeout, 10 MB buffer
- Returns stdout + stderr + exit code

### read_file
- UTF-8 read by absolute path
- Returns file contents or error

### write_file
- Creates parent directories automatically
- Returns byte count written

### glob
- Pattern-based file search
- Returns match list or `(no matches)`

### grep
- Recursive content search with optional `include` glob filter
- Uses ripgrep with grep fallback
- Returns `no matches` on zero results

### ssh
- `execFile`-based — no shell injection
- ControlMaster persistence (`ControlMaster=auto`, `ControlPersist=120`)
- Socket directory restricted to `0700`
- Password sourced from `SSH_PASS` env only — stripped from child process env via `safeEnv()`
- Wake-on-LAN auto-recovery: sends magic packet, polls ping every 3 s for 90 s, retries
- Sudo retry on permission denied

**Acceptance criteria:**
- ✅ bash: executes commands, returns stdout/stderr/exitCode
- ✅ read_file: returns file contents, error on missing file
- ✅ write_file: creates nested dirs, reports byte count, error on unwritable path
- ✅ glob: finds matches, returns `(no matches)` when empty
- ✅ grep: finds content, respects include filter, returns `no matches` when empty
- ✅ ssh: SSH_PASS not leaked to child process env

**Tests:** `tests/tools/bash.test.ts` (5), `tests/tools/filesystem.test.ts` (12)  
**Files:** `src/tools/bash.ts`, `src/tools/filesystem.ts`, `src/tools/ssh.ts`, `src/tools/index.ts`

## F-07 · Ink TUI

**What was built:**  
A live terminal UI using Ink + React. Activates automatically when stdout and stderr are both TTY. Falls back to plain text for pipes and scripts.

**Components:**
- `AgentView` — top-level component, registers all runtime events synchronously before calling `runtime.run()`
- `StepView` — renders one step with status icon, thought, tool name, result preview
- `Spinner` — animated braille spinner via `setInterval`
- `useAgentEvents` — hook for external consumers

**Event handling:**  
All listeners registered in a single `useEffect` before `runtime.run()` is called — eliminates the race between event emission and listener registration. Cleanup uses per-event `runtime.off()` (not `removeAllListeners()`).

**Acceptance criteria:**
- ✅ TUI renders in TTY, plain text in non-TTY
- ✅ Streaming token output shown in real time
- ✅ Per-step status: thinking / executing / done / error
- ✅ `onError` resolves (not rejects) so loop/max-steps errors exit cleanly without unhandled rejection crash
- ✅ Clack menus for `ai chat` interactive input

**Files:** `src/cli/tui/AgentView.tsx`, `src/cli/tui/StepView.tsx`, `src/cli/tui/Spinner.tsx`, `src/cli/output.ts`

## F-08 · Session logging

**What was built:**  
`SessionLogger` writes one markdown file per session to `logDir` when `-v` or `-l` is passed. Each log contains: session metadata (model, CWD, task), each step's thought + tool call + result + exit code, final answer, and any errors.

**Acceptance criteria:**
- ✅ Log file created on session start
- ✅ Each step appended as it completes
- ✅ Final answer recorded with duration
- ✅ Errors recorded in log
- ✅ No log file created when neither `-v` nor `-l` is passed

**Files:** `src/agent/SessionLogger.ts`

## F-09 · OS-aware system prompt

**What was built:**  
`buildSystemPrompt()` injects `process.platform`, `os.release()`, and `os.arch()` into rule #2. The model is told explicitly to use OS-appropriate commands and warned away from Linux-only commands when on macOS.

**Acceptance criteria:**
- ✅ System prompt includes platform, release, and arch at runtime
- ✅ macOS users get `vm_stat`/`sysctl` examples, not `free`/`vmstat`

**Files:** `src/agent/promptBuilder.ts`

**What was built:**
