## Vision
Two clear agents instead of three, with a real plannerтЖТexecutor flow: `plan` designs an approach, then you hand it to `build` with one key.

## Background
`build` (`tools: "all"`) was already a strict superset of `cli` тАФ once file tools learned to expand `~` (the v0.5.1 fix below), `build` could do everything `cli` could. So `cli` was redundant. The missing piece was a way to go from a read-only plan to execution without losing the plan.

## Features

### F-01 ┬╖ Remove the `cli` agent
- Drop `cli` from `BUILTIN_AGENTS` and `BUILTIN_AGENT_IDS`. `build` keeps full shell/infra access (`bash`, `ssh`, `ssh_upload`, `ssh_download`, `fetch`, `download_file`) via `tools: "all"` тАФ nothing transferred, nothing lost.
- `--agent cli` / `/agent cli` now errors `Unknown agent` (the one user-visible behaviour change).
- **Acceptance:** registry lists `["build","plan"]`; `cli` no longer resolves; `build` still permits `bash`.

### F-02 ┬╖ Plan тЖТ build handoff (Tab)
- In the TUI, when the active agent is `plan` and it has produced an approach, pressing **Tab** (outside the slash-command palette) hands off to `build`: switch agent, seed `build` with the plan text, and run it.
- A status-line hint `Tab тЖТ hand off to build` appears when a handoff is available; a notice marks the transition.
- `plan` stays read-only throughout тАФ the handoff is a user-gated transition, so the read-only guarantee holds until you press Tab.
- **Acceptance:** Tab in plan mode with a plan answer switches to `build` and runs it; Tab elsewhere keeps its palette-completion meaning.

### F-03 ┬╖ File tools expand `~` (the prerequisite fix)
- `read_file`/`write_file`/`edit_file`/`append_file`/`delete_file`/`download_file` expand a leading `~` to the home directory (`expandHome`). Fixes `build` "failing" to write to `~/Desktop` (it was writing to `<cwd>/~/Desktop`) тАФ now matches the shell-based `bash` behaviour. (Shipped just ahead of this release.)
